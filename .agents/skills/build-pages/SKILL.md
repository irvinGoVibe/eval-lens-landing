# Build Pages — пакетная сборка внутренних страниц

Превращает продуктовые описания страниц (`wiki/product/<страница>.md` —
полный контент по странице, + `sitemap.md` для маршрутов/структуры, + `vision.md` /
`scope.md` для общего контекста) в живые Next.js-страницы в `web/`, по дизайн-системе
(`wiki/architecture/design-system.md`). Работает **пачкой**: user называет
страницы, skill уходит и собирает их **по очереди, автономно, без прерываний**,
коммитит каждую страницу отдельно — а user потом ревьюит поштучно (каждую story,
каждую страницу, каждый коммит).

Skill **опирается на наших агентов** (`.Codex/agents/`) и переиспользует логику
команд `/story-draft` и `/implement-story`, но осознанно **снимает их
пофазовые гейты подтверждения** ради батч-автономности. Вместо гейтов —
safety-рейлы + коммит на страницу + финальное ревью.

## Чем отличается от обычного flow

Обычный flow (`/story-draft` → `/implement-story` → `/finish-story`) требует
явного подтверждения user на **каждом** переходе фаз. Этот skill — для случая,
когда user заранее согласовал батч и хочет, чтобы всё собралось само.

| | Обычный flow | build-pages |
|---|---|---|
| Гейты между фазами | да, на каждой | нет (один гейт — старт батча) |
| Масштаб | одна story | N страниц за вызов |
| Дизайн-артефакт | — | страница строится сразу по design-system.md |
| Коммиты | вручную/`/finish-story` | по коммиту на страницу |
| Финал | merge worktree | push ветки + сводка на ревью (без merge) |

## Жёсткие safety-рейлы (не отключаются никогда)

Автономность ≠ вседозволенность. Skill при любом раскладе:

1. **Не делает** `git merge`, `git rebase`, `git reset`, не трогает `main`.
   `git push` разрешён — но только текущей рабочей ветки (`eval-lens-landing`),
   только в конце батча и только её коммитов (см. «Финал»). Force-push запрещён.
2. **Не запускает** deploy, установку новых пакетов, миграции к живой БД.
   Команда сборки — `cd web && pnpm build`. **Превью разрешено только для live
   front-end QA** (фаза 5b, `preview_start web` через `.Codex/launch.json`) — это
   единственное исключение из «не поднимать dev-сервер»; если порт 3005 занят
   рабочим сервером user, использовать его или согласовать.
3. **Не правит** `.env*`, `wiki/archive/`, `node_modules/`, `locked`-документы.
4. **Не вводит новые зависимости.** Если страница их требует — фиксирует как
   `blocked`, переходит к следующей странице, рапортует в конце.
5. **Build обязан быть зелёным** перед коммитом каждой страницы. Красный build,
   который не чинится за fix-loop, → страница `blocked`, коммита нет.
6. **Один коммит = одна страница** (её `page.tsx` + компоненты + story-файл +
   правки nav/wiki по этой странице). Не смешивать страницы в одном коммите.
7. Контент берётся **только** из продуктовых источников. Не выдумывать факты о
   продукте; пробелы — в open questions story, не в придуманный маркетинг-текст.
8. **API-ключи я не ввожу.** Если генератор изображений (или иной MCP) требует
   ключ — это ручной шаг user в конфиге MCP; skill ключи не вписывает и не читает.

## Вход

`$ARGUMENTS` — что собрать. Допустимые формы:

- список путей: `/product/entry-hub /trust/methodology /company/about`
- секция sitemap: `все product`, `весь trust`, `company`
- весь P0: `все P0` (список из секции «P0 для первого релиза» в sitemap)
- одна страница: `/trust/security-privacy`

Если `$ARGUMENTS` пуст — показать user список доступных P0-страниц из sitemap и
спросить, какие собрать (через `AskUserQuestion`).

## Phase 0 — Подготовка батча (единственный гейт)

Выполнить **до** какой-либо записи:

1. Прочитать `wiki/product/sitemap.md` (маршруты/структура), `wiki/product/vision.md`,
   `wiki/product/scope.md`, `wiki/architecture/design-system.md`.
2. Развернуть `$ARGUMENTS` в конкретный список маршрутов. Для каждого найти
   **продуктовую страницу** в `wiki/product/` (по slug маршрута, напр.
   `/product/entry-hub` → `wiki/product/entry-hub.md`) — это **первичный источник
   контента и требований к странице** (тексты, секции, какие изображения нужны).
   sitemap даёт маршрут + структурный каркас, vision/scope — общий контекст.
   Если продуктового файла на страницу нет — пометить страницу `blocked`
   («нет контента в `wiki/product/`»), не выдумывать. Дать user список таких.
3. Проверить, какие страницы **уже** существуют в `web/src/app/...` — такие
   пометить «обновление» vs «новая», не перетирать молча.
4. Определить стартовый номер story: max `NN` в `wiki/stories/` + 1.
5. Показать user **план батча** одной таблицей: маршрут · новая/обновление ·
   путь будущего `page.tsx` · `<NN>-<slug>` story · источник контента в sitemap.
6. **Дождаться одного подтверждения** («да» / «поехали» / «го» / «approved»).
   Это **единственный** гейт. После него — автономный прогон без остановок.

Если страниц больше ~8 — предупредить про объём и предложить разбить.

## Per-page pipeline (по очереди, автономно)

Для **каждой** страницы из согласованного списка, строго последовательно:

### 1. Контент
Первичный источник — **продуктовая страница** `wiki/product/<slug>.md` (структура
по `wiki/product/_page-template.md`; парсить по её H2-секциям: «Роль и аудитория»,
«Структура секций», «Контент по секциям», «Числа и факты», «Изображения»,
«Внутренние ссылки», «SEO / meta», «Acceptance»). Из неё берутся тексты, порядок
и типы блоков, числа с источниками и **список требуемых изображений**.
Каркас маршрута/структуры — из sitemap, общий контекст — из `vision.md` /
`scope.md`. Это «продуктовая вкладка», из которой растёт страница. Чего в
источниках нет — в open questions story, не в выдуманный текст.

### 2. Story (агенты `repo-reader` → `story-writer`)
- `repo-reader` — собрать контекст: похожие секции главной для переиспользования
  классов/паттернов, релевантные доки, риски.
- `story-writer` — заполнить **все** секции `wiki/stories/_template.md`.
  Acceptance criteria должны быть проверяемыми и включать:
  - страница доступна по маршруту из sitemap;
  - использует токены/классы из `design-system.md` (никаких новых цветов/теней/
    радиусов/шрифтов);
  - обёрнута в `SiteHeader` + `Footer`, как `web/src/app/page.tsx`;
  - в footer/header добавлена ссылка на страницу (нет «осиротевших» маршрутов);
  - **Apple-grade композиция** (по `page-design-patterns.md`): ≥5 разных
    блок-архетипов, ≥1 pinned multi-screen со scroll-driven появлением, ≥1 тёмный
    full-bleed statement, чередование полос/масштаба — НЕ плоская сетка карточек;
  - движение через data-атрибуты (`ScrollOrchestrator`), без per-section useEffect;
  - `cd web && pnpm build` зелёный;
  - `prefers-reduced-motion` уважается.
- Записать story-файл `wiki/stories/<NN>-<slug>.md`, frontmatter `status: draft`,
  `version: 0.1`, `updated: <сегодня>`, Linear-поля пустыми. **В батче файл
  пишется без отдельного подтверждения** (его заменил гейт Phase 0).

### 3. План (агент `implementation-planner`)
Маппинг acceptance → файлы: где `page.tsx`, какие секции-компоненты под
`web/src/components/sections/` (Server Components), нужны ли стили в `globals.css`,
какие nav-правки. Анимаций по-секционно **не плодить** — если нужен scroll/
параллакс, он идёт в `ScrollOrchestrator.tsx` (правило AGENTS.md). Если план
требует новых пакетов / миграций / правки `locked` — **страница `blocked`**,
переход к следующей, причина в сводку.

### 3.5. Изображения (через MCP-генератор, если подключён)
Если продуктовая страница описывает нужные изображения (hero, иллюстрации,
диаграммы):

1. **Обнаружить генератор** в рантайме через `ToolSearch` (ключевые слова
   «generate image / text-to-image»). Использовать первый подключённый
   image-gen MCP.
2. Для каждого требуемого изображения собрать промпт **в стиле бренда** из
   `design-system.md` (lens-градиент violet→cyan→aqua, Apple-нейтраль, мягкие
   фиолетовые тени, calm — без security-театра) + смысл из продуктовой страницы.
3. Сохранить результат в `web/public/assets/<slug>/<name>.<ext>`, ссылаться как
   `/assets/<slug>/<name>...` (как в `Hero.tsx` / `EvalLenseBentoSection.tsx`).
   Использовать `next/image` для растровых, как в существующем коде.
4. **Если генератора нет** (ToolSearch пусто): не блокировать страницу — поставить
   **видимый размеченный плейсхолдер** `.media-ph` (глобальный примитив в
   `globals.css`): лейбл (что/о чём/соотношение сторон) + hint-промпт + `--ratio`,
   чтобы user **с одного взгляда видел, куда и что вставлять**. Полный промпт — в
   комментарии-шапке `page.tsx`. Список «нужны изображения» — в отчёт страницы.
   Страница остаётся собираемой и коммитится. (НЕ пустой серый div.)
5. **Слотов под визуал должно быть много** (правило Apple-плотности из
   `page-design-patterns.md`): не только hero — диаграммы, визуализация смысла,
   ≥1 горизонтальная галерея. Каждая смысловая секция заслуживает визуала или
   галереи; где ассета нет — `.media-ph`.
6. Изображения — часть коммита **этой** страницы (в `web/public/assets/<slug>/`).
   Бинарники крупнее разумного (напр. >2 МБ) — отметить в отчёте для оптимизации.

### 4. Код (агент `developer-agent`) — Apple-grade, не плоско
Собрать страницу сразу на Next.js по плану, `design-system.md` **и обязательно
`wiki/architecture/page-design-patterns.md`** (каталог блок-архетипов, ритм,
движок движения). Цель — страница уровня Apple product page, **не** сетка
одинаковых `.tcard`.

- маршрут `web/src/app/<path>/page.tsx`, секции инлайн или в `components/sections/`;
- **разные архетипы** из page-design-patterns.md: ≥5–6 разных на 8 секций
  (statement hero · full-bleed statement · pinned multi-screen · editorial split ·
  horizontal gallery · stat band · scrubbed visual · bento · quiet CTA);
- **ритм**: чередовать светлые/тёмные полосы (`.band.ink`), масштаб (full-bleed
  ↔ плотный блок), сторону split-секций; ≥1 тёмный statement и ≥1 pinned
  multi-screen со scroll-driven появлением текста;
- **движение — только через data-атрибуты** (`data-reveal`/`data-scrub`/
  `data-pin`+`data-pin-step`); **смонтируй `<ScrollFX/>` на странице** (после
  `<Footer/>`) — он поднимает их в одном rAF-цикле (примитивы уже в `globals.css`).
  Без `<ScrollFX/>` все `data-reveal` останутся `opacity:0` (пустые секции!).
  НИКАКИХ per-section `useEffect` и сторонних anim-библиотек (правило AGENTS.md);
- **визуальная плотность**: ≥1 горизонтальная галерея + несколько визуальных
  слотов (картинка/диаграмма/визуализация смысла), не текст-стена; числа/процессы/
  сравнения показывать визуально (ring, бары, pipeline, bento). Где реального
  ассета нет — **видимый `.media-ph`** (лейбл + `--ratio` + hint), чтобы было
  ясно, куда вставлять; не пустые серые блоки;
- переиспользовать готовые классы (`.btn`, `.chip`, `.eyebrow`, `.tcard`,
  `.band`, `.wrap`, `.grad-word`, layout-токены `--maxw`/`--gutter`) и
  **канонические** токены `:root`; не `--wf-*`/`--sd-*`; новых цветов/теней/
  радиусов не вводить; lens-акцент — один на экран;
- **рендеринг-гигиена** (заранее, до QA): заголовки через `clamp()`; flex/grid-
  дети с `min-width:0` (иначе текст ломает раскладку); медиа `max-width:100%`;
  без `100vw`/фикс-ширин, дающих горизонтальный скролл; анимировать только
  `transform`/`opacity`; слотам задавать `aspect-ratio` (нет layout-shift);
- кастомные стили страницы — под обёрткой страницы (напр. `.<slug> …`), в конец
  `globals.css`, аккуратно (дерево бывает грязным);
- `prefers-reduced-motion` обязан быть тих (примитивы это уважают; кастом —
  оборачивать в `@media`);
- прогнать `cd web && pnpm build` до зелёного **перед** сдачей.

### 5. QA — статический + **live front-end QA**

**5a. Static review (`ui-reviewer`).** Соответствие дизайн-системе, границы
компонентов, стандарты AGENTS.md, **Apple-grade чек-лист «не плоско ли»** и
**Front-end QA чек-лист** из `page-design-patterns.md` — по чтению CSS/разметки:
заголовки на `clamp()`, `min-width:0` у flex/grid-детей, нет `100vw`/фикс-ширин,
медиа `max-width:100%`, анимация только `transform/opacity`, `aspect-ratio` у
слотов, `<ScrollFX/>` смонтирован, видимые `.media-ph`.

**5b. Live front-end QA (обязательно для этого скилла).** Часть дефектов видна
только в браузере — поэтому здесь **поднимаем превью** (`preview_start` name `web`,
порт 3005) и проверяем вживую. Если 3005 занят рабочим сервером user — использовать
его (страница уже там) либо попросить освободить/разрешить временный порт.
Проверки:
- **Горизонтальный скролл (запрещён):** `preview_eval` →
  `document.documentElement.scrollWidth <= document.documentElement.clientWidth`
  на mobile/tablet/desktop. Превышение — баг (найти переполняющий элемент).
- **Заголовки:** не расползаются и не складываются в столбик по словам/буквам;
  `preview_snapshot` + `preview_screenshot` на разных ширинах.
- **Адаптив:** `preview_resize` mobile(375)/tablet(768)/desktop(1280) — ничего не
  налезает/не обрезается; split→один столбец; pinned-секция не залипает криво.
- **Анимации не отстают:** прокрутить (`preview_eval` scroll), убедиться, что
  reveal/scrub/pin срабатывают вовремя, без рывков; `preview_console_logs` чист
  (нет ошибок/варнингов гидрации).
- **Пустые секции:** убедиться, что контент виден (если `data-reveal` остались
  невидимы — не смонтирован `<ScrollFX/>`).
Снять `preview_screenshot` (desktop + mobile) — пойдут в сводку на ревью.

**5c. Профильные ревьюеры** `data-security-reviewer` / `domain-logic-reviewer` —
только если страница затронула данные/auth/логику (для статики обычно нет).

**5d. `qa-acceptance-reviewer`** — последним: build + acceptance-матрица по story +
агрегирует static- и live-FE-QA findings, выдаёт pass/`needs fix loop`/blocked.

### 6. Fix-loop (автономный, лимит 3)
Если QA вернул провалы (в т.ч. **front-end QA**: горизонтальный скролл,
расползшиеся/столбиком заголовки, налезание на адаптиве, отстающие/дёрганые
анимации, пустые секции, layout-shift) — `developer-agent` чинит **только**
проваленные пункты (`min-width:0`, `clamp()`, `max-width:100%`, убрать `100vw`/
фикс-ширины, перенести анимацию на `transform/opacity`, задать `aspect-ratio`,
смонтировать `<ScrollFX/>`), затем QA (вкл. live-проверку) перезапускается. Не
«замазывать» `overflow-x:hidden` на body — искать причину. До 3 итераций. После
третьего провала — страница `blocked`, её правки **не коммитятся**, переход к
следующей странице.

### 7. Коммит страницы
Только при **зелёном** QA. Один коммит на страницу:
- файлы: `web/src/app/<path>/page.tsx`, новые секции, правки `globals.css`/nav,
  сгенерированные изображения `web/public/assets/<slug>/*`,
  `wiki/stories/<NN>-<slug>.md`;
- сообщение:
  ```
  feat(<path>): build <Page Title> internal page

  Story: wiki/stories/<NN>-<slug>.md
  QA: pass (acceptance N/N)

  Co-Authored-By: Codex Opus 4.8 <noreply@anthropic.com>
  ```
- коммит в **текущую ветку** (`eval-lens-landing`). Push — один раз в конце
  батча (см. «Финал»), не после каждой страницы.

Затем — следующая страница. Сборка одной страницы не блокирует остальные:
ошибка/`blocked` на странице X → записать причину, идти к X+1.

## Финал батча — push + сводка на ревью

После всех страниц:

1. **Push** текущей ветки: `git push -u origin eval-lens-landing` (без
   force, без merge). Если ветки нет на remote — создаётся upstream. Если push
   отклонён (расхождение с remote) — **не** делать force/rebase: вывести ошибку и
   отдать решение user. Все per-page коммиты уходят разом — разбираете по коммитам.
2. Вывести **сводную таблицу** (без merge):

| Страница | Story | Коммит | QA | Acceptance | На что смотреть |
|---|---|---|---|---|---|
| /product/entry-hub | 04-... | `a1b2c3d` | pass | 6/6 | hero-числа, ring |
| /trust/methodology | 05-... | `e4f5g6h` | pass | 5/5 | judge-roles блок |
| /company/about | — | — | **blocked** | 3/5 | нужен новый пакет X |

Плюс **per-page чек-лист ручного ревью** (для каждой собранной страницы):

1. `git show <commit>` — диф по странице.
2. Посмотреть **скриншоты из live-FE-QA** (desktop + mobile, фаза 5b) в сводке;
   при желании открыть превью самому (`preview_start web` / свой сервер на 3005).
3. Прогнать визуальный `design-critique` (skill `design:design-critique`) по
   скриншоту страницы — UX, иерархия, соответствие стайл-гайду.
4. Проверить сгенерированные изображения (или список «нужны изображения», если
   генератор был не подключён) — при необходимости перегенерить/подложить.
5. При ОК — повысить story `status` вручную (skill сам не повышает).
6. PR из ветки `eval-lens-landing` — вручную, когда батч устраивает (ветка уже
   запушена на финале батча).

Для `blocked`-страниц — явный список, что разблокирует (согласовать пакет,
дописать продуктовый контент, расширить scope).

## Чего skill не делает

- не переходит к коммиту, если build красный или QA `blocked`;
- не делает merge / rebase / force-push / deploy / установку пакетов / миграции
  (обычный `git push` рабочей ветки в конце батча — разрешён);
- не вписывает и не читает API-ключи (подключение MCP-генератора — шаг user);
- не повышает `status` story до `review`/`approved`/`locked` — это user;
- не запускает dev-сервер, кроме **live front-end QA** (фаза 5b, `preview_start web`);
- не выдумывает продуктовый контент сверх sitemap/vision/scope;
- не дробит scroll-анимации по `useEffect` — всё в `ScrollOrchestrator.tsx`;
- не вызывает Linear MCP (трекинг — `/linear-publish` отдельно, если нужен);
- не смешивает несколько страниц в одном коммите.

## Если что-то пошло не так

Любая страница может упасть, не разблокировав остальные. Но если ломается
**общее** (например, `pnpm build` падает на уже закоммиченной странице и красит
весь проект) — **остановись**, верни user текущее состояние батча (что
закоммичено, что нет) и жди инструкций. Не «проталкивай» батч ради галочки.
