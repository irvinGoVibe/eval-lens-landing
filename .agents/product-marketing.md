### История бренда (ключевой нарративный актив)
Продукт начинали как **AI Jury** на **Amazon Nova hackathon** — идея: несколько специализированных AI-судей вместо одного общего мнения модели. После **сотен прогонов** обнаружили: больше судей ≠ лучше результат (баллы плавали между прогонами, судьи повторяли друг друга, длинные отчёты давали шум вместо ясности). Это сменило направление — от «набора AI-агентов» к **управляемой системе оценки**. По мере работы пришли к фундаментальному выводу:

> **AI не должен судить. Судит и принимает финальное решение — человек. AI готовит материал для этого решения.**

Из этого тезиса вырос ребрендинг в **EvalLense** и более зрелый, понятный продукт: не «AI-судья, выносящий вердикт», а **линза** (lens), через которую человек видит каждую заявку структурированно, с доказательствами, и принимает решение быстрее и увереннее. Этот переход «от AI Jury к EvalLense» — сама по себе сильная история для контента и позиционирования.

## 1. Обзор продукта (Product Overview)
- **One-liner:** «Lens Your Next Unicorn» — операционный слой для пакетной оценки питч-деков, где AI готовит доказательный анализ, а финальное решение принимает человек.
- **Что делает:** Batch-review питч-деков: организатор собирает заявки в одном месте, AI-панель оценивает каждую по единым критериям с привязкой к доказательствам из дека, выдаёт структурированные отчёты и вопросы для live Q&A. Организатор изучает отчёты и **сам ставит финальные баллы**; лидерборд строится по человеческим оценкам.
- **Категория (полка):** decision-support / evaluation platform для отбора заявок. Клиент ищет это как: «pitch competition software», «AI-powered judging / screening», «hackathon judging», «accelerator application review», «deal flow screening».
- **Тип продукта:** B2B SaaS (веб-приложение + API).
- **Бизнес-модель:** управляемый сервис; монетизация (тикеты/подписка) — Фаза 2 роадмапа, post-MVP.

## 2. Целевая аудитория (Target Audience)
- **Целевые организации:** организаторы питч-конкурсов, хакатонов, акселераторов и инкубаторов, VC-фонды (скрининг дилфлоу), грантовые программы, университеты, исследовательские отборы.
- **Лицо, принимающее решение:** **Organizer** (batch owner) — тот, кто получает поток заявок и должен их отранжировать. Это основной и единственный авторизованный пользователь MVP.
- **Основной use case:** быстро и объективно оценить десятки–сотни питч-деков и выбрать сильнейших, не теряя дни на ручной разбор.
- **Jobs to be done:**
  - «Прогнать пакет заявок и поднять наверх сильнейших, не читая каждый дек по часу».
  - «Получить по каждой заявке доказательный разбор и вопросы для live-питчинга».
  - «Принять обоснованное человеческое решение и оставить аудиторский след».
- **Участники (Participants)** в систему не входят: только загружают дек (self-upload) и получают фидбек через организатора.
- **Масштаб:** от 10–20 заявок (small pitch) до сотен (массовые конкурсы).

## 4. Проблемы и боли (Problems & Pain Points)
- **Ядро проблемы:** организатор зажат между **временем** и **объективностью**. Внимательный разбор одной заявки — 1–3 часа; на десятках/сотнях процесс растягивается на недели и срезается до поверхностного скана.
- **Почему альтернативы не вывозят:**
  - Ручная оценка — медленно и субъективно (биасы, усталость, разные шкалы; два честных эксперта расходятся на 3 балла из 10).
  - Простые «загрузи дек → AI-фидбек» инструменты — чёрный ящик, невоспроизводимо, один общий промпт без методологии.
- **Что это стоит:** недели времени, упущенные сильные заявки, решения, которые трудно защитить.
- **Эмоциональное напряжение:** страх пропустить «единорога», недоверие к «AI, который решает за тебя», давление сроков конкурса.

## 5. Конкурентное поле (Competitive Landscape)
- **Прямые:** специализированные платформы для питч-конкурсов / event-management с функцией оценки. Чаще без доказательной AI-методологии.
- **Вторичные:** «AI-обёртки» (OpenAI API + промпт) для разбора дека — быстро, но чёрный ящик и невоспроизводимо.
- **Косвенные:** Notion/Excel-рубрики и ручные процессы жюри — прозрачно, но не масштабируется.
- *Явного конкурентного анализа в доках пока нет — раздел уточнить с фаундером.*

## 6. Дифференциация (Differentiation)
- **Нет чёрного ящика:** каждый балл разложен по критериям, каждый критерий — на конкретные сигналы с **цитатами из дека** (explainability через SourceRef).
- **Воспроизводимость:** один и тот же дек даёт идентичный результат (детерминированный scoring-слой, фиксированные JSON-контракты).
- **Методологическая послойность:** не один промпт «оцени всё», а **панель из 6 специализированных судей** (P1–P6), честная агрегация и детекция расхождений (Spread).
- **Human-in-the-loop by design:** AI готовит материал — человек решает. Лидерборд — по человеческим Jury Score, не по AI Total Score.
- **Настройка под формат:** Pitch Competition и Hackathon — разные жанры с разными критериями; Custom — post-MVP.
- **Почему выбирают нас:** скорость без потери доверия — «informed decision за минуты вместо часов», с прозрачностью и контролем.

## 8. Динамика переключения (Switching Dynamics — JTBD Four Forces)
- **Push:** недели ручного разбора, субъективность, невозможность защитить решение.
- **Pull:** пакетный доказательный анализ за минуты + контроль финального решения.
- **Habit:** привычные Excel/Notion-рубрики и «прочитаю все деки сам».
- **Anxiety:** «AI решит за меня и ошибётся» → снимается тезисом human-in-the-loop и прозрачностью.

## Приложение: как работает продукт (для точности в текстах)

Пайплайн: **Decoder** (любой формат → PDF → структурный JSON по слайдам, флаг `pitch_available`) → **Judge Panel** — 6 судей параллельно и независимо:
- **J-P1 Problem Significance** · **J-P2 Solution Differentiation** · **J-P3 Business Value / Market** · **J-P4 Pitch Quality** · **J-P5 Team Readiness** · **J-P6 Feasibility**

→ **Summarizer**: *Function 1* (детерминированная математика → AI Total Score 0–10, advisory) + *Function 2* (нарратив: strengths/weaknesses, why-can-pass/fail, deck completeness, вопросы для участников) → **AI Report** (read-only для организатора) → **Human Review**: организатор ставит **Jury Score** → **Leaderboard** по Jury Score.

**Канонические термины для точности:** `AI Total Score` (advisory reference, не для ранжирования) vs `Jury Score` (финальный человеческий балл, идёт в лидерборд). Spread(d) — расхождение судей (≥3.0 = конфликт, подсвечивается).

**Стек (для контекста, не для внешних текстов):** продукт — Next.js + Supabase + Vercel AI Gateway. Лендинг — Next.js 16 + React 19 + Tailwind v4 + Three.js (3D-сцена с единорогом), деплой Vercel.
