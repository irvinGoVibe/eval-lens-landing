Ты — исполнитель команды `/implement-story` из EvalLense Landing. Оркестрируешь полный цикл реализации одобренной story: план → код → QA → fix-loop → финальный отчёт. Полный контекст — `PROJECT-ENTRYPOINT.md`, правила реализации — `CLAUDE.md`, описание агентского flow — `wiki/processes/agent-workflow.md`.

## Вход

`$ARGUMENTS` — путь к story в `wiki/stories/`, например `wiki/stories/04-feedback-form.md`.

## Жёсткие гейты

Команда **никогда** не переходит между фазами без явного подтверждения user. Подтверждение — одно из: `да`, `ок`, `подтверждаю`, `code`, `qa`, `retry`, `done`.

Фазы:
1. plan
2. code
3. qa
4. fix-loop (опционально, до 3 итераций)
5. final report

## Linear auto-transitions

**Linear-guard:** если в `PROJECT-ENTRYPOINT.md` секция Linear помечена как «не подключён» — пропусти все Linear-переходы в этой команде, выполни остальную логику и напиши в отчёты фаз «Linear не подключён, переходы пропущены».

Подтверждение фазы **= согласие на соответствующий Linear-переход**. Команда сама вызывает `mcp__*__save_issue` с новым state, без отдельного вопроса. Маппинг (см. `wiki/processes/linear-workflow.md`):

| Триггер | Linear переход |
|---|---|
| подтверждение Phase 2 (start code) | привязанные Issues: `Todo → In Progress` |
| Phase 5 запуск (после QA pass) | привязанные Issues: `In Progress → In Review` |
| Phase 4 fix-loop стартует | Issues остаются в `In Progress` (статус не двигаем) |
| Phase 3 QA `blocked` без user-фикса | Issues остаются в `In Progress`, не откатываем |

Откуда берём список Issues — из frontmatter story (`linear_issues:`). Если поле пустое — переходы не делаем, пишем в отчёт «Linear не привязан к story, переходы пропущены». **Done не ставим никогда** — это вручную user в Linear UI.

Если переход не удался (Linear API отказ, статус не существует в команде, issue не найден) — пишем ошибку в отчёт фазы, фаза идёт дальше.

## Что делаешь

### Фаза 1 — Plan

1. Прочитай story по пути `$ARGUMENTS`. Если файла нет, или `Status:` не `Draft`/`Approved`, или не заполнены acceptance criteria — **остановись** и опиши user, что не так.
2. Запусти `implementation-planner`. Передай ему путь story.
3. Покажи user план в формате, который выдал planner: acceptance → code map, files likely to change, migrations, safety checks, risks, rollback notes.
4. **Жди подтверждения** для перехода в фазу code.

Если в плане затронуты:
- `locked` документы — **остановись**, покажи предупреждение и явно спроси «Подтверди, что действительно хочешь редактировать `locked` документ»
- новые пакеты/зависимости — **остановись**, спроси, согласовано ли
- изменение схемы БД без миграции — **остановись**, объясни, что нужна миграция

### Фаза 2 — Code

1. **Сразу после подтверждения** (до запуска `developer-agent`) — выполни Linear-переход: для каждого ID из `linear_issues` story сделай `mcp__*__save_issue` с `state = In Progress`. Если поле пустое или Linear не подключён — пропусти. Результат каждого перехода логируется в отчёт фазы.
2. Запусти `developer-agent` с входом: путь story + план от planner.
3. Дождись отчёта `developer-agent`: что реализовано, что нет, изменённые файлы, build status.
4. Если `cd web && pnpm build` упал — `developer-agent` обязан фиксить до зелёной сборки прежде, чем сдавать отчёт.
5. **Не переходи в QA автоматически.** Покажи user отчёт `developer-agent` (включая результат Linear-переходов из шага 1) и жди подтверждения.

### Фаза 3 — QA

1. По диффу определи, какие профильные ревьюеры нужны:
   - затронуты схема/миграции БД, аутентификация, storage, middleware, работа с секретами → запусти `data-security-reviewer`
   - затронута доменная логика: core business rules, серверные пайплайны, API-роуты → запусти `domain-logic-reviewer`
   - затронуты страницы, компоненты, хуки, клиентский UI → запусти `ui-reviewer`
2. Запусти `qa-acceptance-reviewer` последним — он агрегирует выводы профильных ревьюеров и формирует acceptance-матрицу.
3. Покажи user итоговую матрицу.

### Фаза 4 — Fix loop

Запускается, только если QA вернул `needs fix loop` или `blocked` с конкретными проваленными пунктами.

Лимит: **3 итерации**. После третьей провальной итерации — **остановись**, отдай user полный список проваленных пунктов и не запускай ещё одну.

Каждая итерация:
1. Покажи user список фиксов от `qa-acceptance-reviewer`.
2. Жди подтверждения `retry`.
3. Запусти `developer-agent` **только** на проваленные пункты. Никаких новых правок.
4. Перезапусти QA-фазу (профильные ревьюеры по диффу + `qa-acceptance-reviewer`).
5. Если pass — переходи в фазу final report. Если fail — следующая итерация (если бюджет не исчерпан).

### Фаза 5 — Final report

1. **До запуска `release-reporter`** — выполни Linear-переход: для каждого ID из `linear_issues` story сделай `mcp__*__save_issue` с `state = In Review`. Если поле пустое или Linear не подключён — пропусти. Результат каждого перехода вставь в финальный отчёт.
2. Запусти `release-reporter` с входом: story, план, отчёт `developer-agent`, отчёты ревьюеров, финальный отчёт `qa-acceptance-reviewer`.
3. Покажи user финальный отчёт. Явно укажи: «Issues переведены в In Review. Done — ставь вручную в Linear UI после ручной проверки» (если Linear подключён).
4. **В конце финального отчёта** добавь явное напоминание: «Story реализована и отрапортована. Чтобы замержить worktree в eval-lens-landing и закрыть story, запусти `/finish-story <путь-к-story>`». Сам `/implement-story` git-команды **не делает** — merge и cleanup worktree — отдельный шаг через `/finish-story`.
5. Не правь story, wiki, notes — `release-reporter` только перечисляет, что user обновляет вручную или отдельной командой. Linear-переход в `In Review` — единственное автоматическое действие на этой фазе.

## Чего не делаешь

- не переходишь между фазами без явного подтверждения user
- не запускаешь `developer-agent` без плана от `implementation-planner`
- не запускаешь `release-reporter`, если QA `blocked`
- не делаешь больше 3 итераций fix-loop
- не **создаёшь** новые Linear Issues и не **удаляешь** Issues — это `/linear-publish` и `/note` / `/task`. Только переходы между статусами на Phase 2 и Phase 5.
- не ставишь Issue в `Done` — `Done` ставит **только** user вручную в Linear UI
- не делаешь `git commit`, `git push`, не запускаешь dev-сервер, не устанавливаешь пакеты, не деплоишь, не применяешь миграции к живой БД
- не правишь `.env*`, `wiki/archive/`, `node_modules/`
- не правишь `locked` документы без явного подтверждения

## Если что-то пошло не так

В любой момент при ошибке, конфликте плана и кода, упавшей сборке, которую не получается починить — **остановись**, верни user текущее состояние и жди инструкций. Не «проталкивай» цикл вперёд, чтобы «закрыть» story.
