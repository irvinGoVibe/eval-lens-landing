Ты — исполнитель команды `/story-draft` из EvalLense Landing. Готовишь story-черновик из сырого сигнала. Полный контекст — `PROJECT-ENTRYPOINT.md`. Связанные процессы — `wiki/processes/process.md`, `wiki/processes/linear-workflow.md`, `wiki/processes/agent-workflow.md`.

**Linear-guard:** если в `PROJECT-ENTRYPOINT.md` секция Linear помечена как «не подключён» — пропусти шаг 7 (Linear-переходы), выполни остальную логику и сообщи об этом пользователю.

## Вход

`$ARGUMENTS` — описание задачи, цитата из чата или ссылка на источник (`notes/...`, `wiki/...`).

## Что делаешь

1. **Запусти `repo-reader`** с входом `$ARGUMENTS`. Получи отчёт: source files, relevant docs, affected areas, missing context, risks.

2. **Запусти `story-writer`** с входом:
   - текст задачи `$ARGUMENTS`
   - отчёт `repo-reader`

   `story-writer` обязан читать `wiki/stories/_template.md` каждый запуск и заполнить **все** секции шаблона. На этом этапе файл не создаётся.

3. **Переименуй текущий Claude Code чат** в `<NN>-<slug>` (slug story из `story-writer`).
   - формат: `<NN>-<slug>` или `story-<NN>-<slug>` — короткое, без пробелов
   - используй встроенный механизм Claude Code для смены названия сессии (`/title <NN>-<slug>` или эквивалент в текущем окружении)
   - если программного способа нет — выведи в чат предлагаемое название и попроси user переименовать вручную; не блокируй на этом шаге
   - переименование делаешь **до** показа черновика, чтобы чат сразу был идентифицируем по story

4. **Покажи user в чате:**
   - предлагаемый путь `wiki/stories/<NN>-<slug>.md`
   - required approvals
   - open questions
   - suggested Linear mapping (team + mode)
   - **полный markdown story-черновика**

5. **Жди явного подтверждения user.** Подтверждение — одно из: `да`, `ок`, `approved`, `публикуй`, `создавай`, `пушь`. Молчаливое продолжение **не считается** подтверждением.

6. **После подтверждения:**
   - проверь, что путь свободен; если занят — спроси user, переименовать или перезаписать
   - запиши файл с frontmatter: `status: draft`, `version: 0.1`, `updated: <YYYY-MM-DD>` — story-черновик всегда создаётся в `draft`
   - **frontmatter Linear-полей** (`linear_project`, `linear_issues`) оставляй пустыми, если user не назвал явно ID. Не выдумывай.
   - в шапке story — `Status: Draft`
   - не повышай статус сам (никогда `review` / `approved` / `locked` — их ставит только человек)
   - если на шаге 3 переименование чата не удалось — после записи файла повтори попытку (теперь slug точно финальный)

7. **Linear-переход после записи (опционально):**
   - если user указал ID существующего Issue из — (или их список) — впиши их в frontmatter `linear_issues:` и предложи перевести каждый из `Backlog` в `Todo` через `mcp__*__save_issue` (state-обновление)
   - если ID-ов нет — никаких переходов не делаешь. Linear синхронизирует `/linear-publish` на следующем шаге
   - переход делаешь только после явного «да» user. Команда `/story-draft` сама ничего не двигает молча.

8. **Отчёт user.** После записи верни:
   - путь к файлу
   - какие Linear-переходы выполнены (если были) — формат: `<identifier>: Backlog → Todo`
   - предложение следующего шага: `/linear-publish <путь>` (если Linear ещё пустой) или `/implement-story <путь>` (или ничего, если user пока не готов)

## Чего не делаешь

- не пишешь файл до явного подтверждения user
- не правишь существующие story
- не правишь `wiki/stories/_template.md`
- не публикуешь в Linear (для этого есть `/linear-publish`)
- не вызываешь `mcp__*__save_*` (кроме опционального шага 7 после явного «да»)
- не запускаешь `developer-agent` или `implementation-planner`
- не правишь `wiki/` (кроме записи новой story в `wiki/stories/`), `notes/`, `src/`, конфигурацию, `.env*`
- не отступаешь от структуры `_template.md`

## Если контекста не хватает

Если по итогам `repo-reader` видно, что:
- ключевая логика не зафиксирована в `wiki/`
- acceptance невозможно сформулировать как проверяемые
- scope явно плавает

— **остановись** и предложи user сначала зафиксировать контекст в `notes/` через `/note`, а после — повторить `/story-draft`.
