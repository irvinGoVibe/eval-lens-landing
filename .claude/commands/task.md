Ты — исполнитель команды `/task` из EvalLense Landing. В отличие от `/note`, не пишет в `notes/daily/` и используется, когда задача родилась прямо из обсуждения и нужна сразу как Linear Issue. Полный контекст — `PROJECT-ENTRYPOINT.md`. Маршрутизация — `wiki/processes/linear-workflow.md`.

**Linear-guard:** если в `PROJECT-ENTRYPOINT.md` секция Linear помечена как «не подключён» — Linear-шаги выполнять не во что. Сообщи user, что Linear не подключён, и предложи вместо этого зафиксировать задачу через `/note` в `notes/daily/`.

## Когда использовать

- задача обсуждена в чате, в notes писать нечего
- надо быстро накидать набор задач в Backlog (можно вызывать команду подряд)
- ещё нет story — просто фиксируем сигнал

## Когда **не** использовать

- если уже есть story → используй `/linear-publish <story>`
- если задача уже опубликована и её надо двигать → это делает `/story-draft` или `/implement-story`, не `/task`
- если нужно зафиксировать сырой сигнал в notes → `/note`

## Что делаешь

1. Разбери префикс в начале `$ARGUMENTS`:

   | Префикс | Команда | Стартовый статус | Тип |
   |---|---|---|---|
   | `bug:` | — | `Backlog` | Bug |
   | `task:` или нет префикса | — | `Backlog` | Task |
   | `idea:` | — | `Inbox` | Idea |

2. Сформируй короткий план Issue:

   - title — текст без префикса, обрежь до разумной длины
   - description — одна строка контекста («создано через `/task` из обсуждения, <YYYY-MM-DD HH:MM>») плюс полный текст из `$ARGUMENTS`. Никаких ссылок на notes — их нет.
   - team / state по таблице выше
   - labels — пусто, если в тексте явно не указано
   - estimate — пусто

3. Покажи user план одной короткой строкой и **жди явного «да»**. Допустимые подтверждения: `да`, `ок`, `создавай`, `пушь`, `публикуй`.

4. После подтверждения создай Issue через `mcp__*__save_issue`. Верни `identifier` и URL.

5. Если user сразу подаёт следующую задачу через `/task` — обрабатывай как новую сессию: показал план → дождался «да» → создал. Не накапливай очередь без подтверждений.

## Множественные задачи

Если в одном вызове передан список (например, маркированный список), отнесись к нему как к одному действию:
- покажи user сводную таблицу `| # | team | state | title |`
- жди одно общее «да»
- после подтверждения создавай Issues по одному, но без повторных подтверждений
- если хоть один `save_issue` упал — остановись, верни user, что создалось, что нет

## Чего не делаешь

- не пишешь в `notes/`
- не создаёшь Linear Issue без явного подтверждения
- не двигаешь Issue в `Todo` / `In Progress` / `In Review` (это `/story-draft` и `/implement-story`)
- не ставишь Issue в `Done` — только user вручную
- не привязываешь Issue к story (нет story на этом этапе) и не редактируешь `wiki/stories/`
- не создаёшь Linear Project — для проекта используется `/linear-publish` на основе story

## Аргументы

`$ARGUMENTS` — текст задачи, опционально с префиксом, либо несколько задач списком.
