# Rule: contract-lock (неизменяемый контракт блока — Фаза 1)

**Источник правды:** контент-инвариант redraw-block + `surface-invariant`.

Перед дизайном фиксируется контракт, который **одинаков во всех V1/V2/V3** и не
ломается реализацией. Три части.

## 1. Semantic contract — один и тот же смысловой payload в V1/V2/V3

Сохраняются: heading, eyebrow, body copy, CTA labels, ссылки, карточки, метрики,
числа, данные, **порядок смыслов**, функциональная цель, смысл медиа, все content slots.

- Нельзя давать разным версиям **разные тексты**, чтобы одна выглядела лучше.
- Нельзя сокращать контент ради красивой композиции.
- Нельзя придумывать **новые продуктовые утверждения**.

## 2. Functional contract

Сохраняются: props, структура данных, ссылки, routing, смысл интеракций,
CMS-контракт, production-поведение, серверность.
- Если утверждённый дизайн требует нарушить функциональный контракт → вернуть
  **`design-conflict`** и остановиться (не ломать молча).

## 3. Theme contract (внутри одной версии)

Light и Dark одной версии **обязаны совпадать** по: DOM-структуре, layout, grid,
размерам, spacing, type scale, line-height, letter-spacing, max-width текста,
media ratio, размещению CTA, высоте секции, структуре интеракций, motion-контракту.
Меняется **только цвет** (см. [surface-invariant](surface-invariant.md)).

Сравниваются попарно: **V1 Light ↔ V1 Dark**, **V2 L↔D**, **V3 L↔D**.
Геометрия V1/V2/V3 между собой **может и должна** отличаться. При смене темы текст
не должен прыгать или менять ширину.

## Выход фазы (оркестратор фиксирует)

```
## Contract lock
- Semantic slots (инвариант): …
- Functional contract: props/data/routing/serverness …
- Theme contract: что обязано совпадать L↔D …
- Конфликты/риски: …
```
