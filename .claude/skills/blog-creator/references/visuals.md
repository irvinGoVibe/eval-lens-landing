# Визуал статьи — картинки, таблицы, интерфейсы

Скилл не только пишет текст: для каждого блока, где визуал усилит мысль, он
**проектирует варианты и предлагает пользователю выбор** (по умолчанию **3
варианта на слот**). Пользователь выбирает — мы фиксируем путь и (для картинок)
готовый промпт под генератор.

> **Принцип.** Визуал ставим там, где он несёт смысл (доказательство, структура,
> сравнение, шаг пайплайна), а не для украшения. Каждый слот привязан к
> конкретной секции статьи и к её тезису.

---

## Шаг V0. Посчитай, сколько визуалов нужно под объём статьи

Бюджет визуала идёт из плана (Шаг 2 скилла) и калибруется по объёму, чтобы статью
**раскрыть**, а не задекорировать. Ориентиры:

- **Cover** — всегда 1.
- **Плотность тела** — ~**1 визуал на каждые 200–350 слов** ИЛИ на каждую крупную
  H2-секцию (что больше). Грубо по объёму:
  | readMinutes | ~слов | inline-визуалов (без cover) |
  |---|---|---|
  | 2–3 | 400–700 | 1–2 |
  | 4–5 | 800–1200 | 2–4 |
  | 6–8 | 1300–2000 | 4–6 |
  | 9+ | 2000+ | 6–8 |
- **Обязательный визуал на «тяжёлый» блок** независимо от плотности: пайплайн/поток,
  сравнение/таблица критериев, метрики/числа, экран продукта (UI). Если в секции
  есть такой блок — он получает слот.
- **Не недогружать** (стена текста без воздуха) и **не перегружать** (визуал ради
  визуала). Если слотов выходит больше бюджета — оставь те, что несут доказательство
  или структуру, остальное в текст.

Зафиксируй итог: «cover + N визуалов (типы: …)» — это и есть бюджет для V1–V2.

## Шаг V1. Найди визуальные слоты

Пройди по готовому тексту и отметь места, где визуал работает лучше абзаца:
- **Cover** — обложка статьи (обязательно, 1 шт).
- **Inline-картинки** — там, где нужен образ/сцена/метафора режима.
- **Диаграмма/схема** — пайплайн, поток, архитектура, «было → стало».
- **Таблица** — сравнение, критерии, P1–P6, бенчмарки, pros/cons.
- **Интерфейс/UI-мокап** — как выглядит отчёт/лидерборд/Review Board/Entry Hub.

Для каждого слота зафиксируй: id-слота, секция, что доказывает, рекомендуемый
тип, соотношение сторон (cover 16:9; inline по месту — 16:9 / 4:3 / 1:1).

## Шаг V2. Дай 3 варианта на слот

Для каждого слота предложи **3 варианта** в порядке рекомендации. Формат варианта:

```
Слот: <id> — <секция> (тип: image | diagram | table | interface; ratio)
  Вариант A (рекомендую): <короткое название>
    — Что показывает: <1 строка>
    — Почему подходит: <связь с тезисом секции>
    — Реализация: <промпт для генератора | разметка таблицы | описание UI-мокапа>
    — Путь: /assets/blog/<short>/<slot>.<ext>
  Вариант B: …
  Вариант C: …
```

Варианты должны быть **разными по подходу**, а не косметическими правками одного
(например: абстрактная метафора / буквальная диаграмма / реальный UI-мокап).

## Шаг V3. Реализация по типам

### Картинка (image / cover / метафора)
Готовый промпт в **фирменном стиле EvalLense** (см. память
`evallense-page-media-slots` и `globals.css`-токены):
- **Акцент — только lens-градиент** `118° violet #6c4cf1 → lavender #a99bff →
  cyan #2ec5e8 → aqua #36e0c2` (не радуга).
- Hairline 1px структура, много воздуха, soft violet depth, Apple-neutral sans +
  mono-метки.
- Фон на выбор: **белый** (`#ffffff`/`#f5f5f7`, текст `#1d1d1f`) или **чёрный**
  (`#0a0a0d→#1b1b26`, текст `#f5f5f7`).
- Статус-акценты при нужде: green `#1aa37a`, amber `#e8943a`.
- **Единорог — наш фирменный маскот**, используем свободно: рендерим премиально
  (кристалл/стекло/low-poly) в lens-градиенте, не детский мультяшный стиль. Может
  быть героем обложки или акцент-элементом.
- **Нельзя:** security-theatre/щиты, 3D-хром, стоковые люди-фото, чужие логотипы.
- **Числа/текст в визуале:** реальные метрики не выдумывать; иллюстративные
  placeholder-числа/подписи (напр. демо-балл «7.3») — ок, если не выдаются за
  настоящие данные.

Промпт пиши готовым к вставке в генератор (англ.), 1–3 предложения + стиль-хвост.

### Диаграмма / схема
Предложи варианты: (а) ASCII/```text```-блок прямо в Markdown (как пайплайн
`Deck → Decoder → … → Leaderboard` в эталонной статье) — самый дешёвый и
редактируемый; (б) сгенерированная картинка-схема в фирменном стиле; (в)
mermaid-подобное описание, если нужно. Для пайплайна по умолчанию рекомендуй (а).

### Таблица
Дай **3 варианта раскладки** одних и тех же данных, напр.:
- A — компактная (2–3 колонки, суть);
- B — с колонкой «почему важно / что ломается»;
- C — сравнительная «EvalLense vs обычный процесс» / pros-cons.
Помни про AEO: таблица должна читаться как самодостаточный ответ. Markdown-таблица
по умолчанию (рендерится в теле), картинку-таблицу — только если так нагляднее.

### Интерфейс / UI-мокап
Опиши, что на экране (а не просто «скриншот продукта»): какой экран
(evidence-based report / leaderboard / Review Board / Entry Hub), какие элементы
видны (P1–P6, AI Total Score *advisory*, Jury Score, SourceRef, confidence),
где видно **human-in-the-loop**. Дай 3 ракурса: (A) полный экран-обзор; (B) крупный
фрагмент с одним доказательством (SourceRef → слайд); (C) «до/после» решения
человека. Стиль мокапа — те же фирменные токены; реальные данные не выдумывать,
числа — placeholder или из источников правды.

## Шаг V4. Зафиксируй в статье и отчёте

- В теле статьи ставь слот по конвенции: `:::gallery ![](...) :::` или таблицу
  прямо в Markdown; cover — в frontmatter `cover:`.
- **Конвенция путей:** `web/public/assets/blog/<short>/<slot>.<ext>` (в копии —
  `/assets/blog/<short>/<slot>.<ext>`), где `<short>` — короткий ключ статьи
  (как `how-evallense-evaluates`, `whats-next`).
- В финальном отчёте перечисли все слоты с **3 вариантами** и пометкой
  «рекомендую», чтобы пользователь выбрал. После выбора — оставь только выбранный
  вариант с готовым промптом/разметкой и путём.
