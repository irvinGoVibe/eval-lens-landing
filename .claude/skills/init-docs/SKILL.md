# Init Docs — интервью и первичная документация

Превращает пустой проектный контейнер в задокументированный: задаёт серию вопросов о продукте и архитектуре, показывает черновики и после подтверждения создаёт стартовый набор документов.

## Жёсткие правила

1. **Не выдумывать.** В документы попадает только то, что сказал пользователь. Всё неотвеченное — в секцию «Открытые вопросы» соответствующего документа, не в придуманные формулировки.
2. **Не генерировать backlog.** Skill не создаёт stories, задачи и feature-списки — для этого есть `/story-draft` и `/note`. Только документация.
3. Все создаваемые wiki-документы — `status: generated`, `version: 1.0`, `updated: <сегодня>` (AI-output, ждёт ревью человеком — правило из `wiki/processes/process.md` §3).
4. Запись файлов — только после показа черновиков и явного подтверждения.
5. Язык документов — язык проекта из `PROJECT-ENTRYPOINT.md`.
6. Перед стартом прочитать `PROJECT-ENTRYPOINT.md` и `CLAUDE.md` — часть ответов (имя, описание, стек) уже есть после bootstrap; не переспрашивать известное, а предлагать как дефолт для уточнения.

## Phase 0 — Repo scan (если в репо уже есть код)

Если проект brownfield (в репо есть код: package.json, src/, конфиги) — сначала просканировать факты, потом спрашивать:

1. Стек по факту: манифесты пакетов, конфиги, lock-файлы
2. Структура: маршруты/страницы/модули, точки входа
3. Данные: модели, миграции, схемы
4. Интеграции: env-переменные, SDK во внешние сервисы

Интервью после этого — **подтверждающее**: «я вижу X — верно?» вместо «расскажите с нуля». Архитектурный документ собирается из фактического кода; расхождения между кодом и словами пользователя помечаются явно в «Открытых вопросах», молча не разрешаются.

5. Obsidian file-hider: дописать обнаруженные на этом шаге верхнеуровневые code-папки проекта в `.obsidian/plugins/OA-file-hider/data.json` → `hiddenList` (чтобы в проводнике Obsidian остались видны только `Home.md`, `notes/`, `wiki/`). Существующие записи не удалять, только добавлять отсутствующие.

## Phase 1 — Интервью

Вопросы задаются через `AskUserQuestion` батчами (до 4 за раз); у каждого вопроса есть варианты-подсказки + «Other» для свободного ответа. Пользователь может ответить «пропустить» на любой блок — тогда блок фиксируется как открытые вопросы.

**Блок 1 — Продукт:**
- Что строим? (одно-два предложения сути)
- Для кого? (аудитория, роли пользователей)
- Какую боль решает? Почему существующие решения не подходят?
- Главная ценность / отличие?

**Блок 2 — Сценарии и scope:**
- Главный пользовательский flow от входа до ценности (3–7 шагов)?
- Второстепенные сценарии, которые важны?
- Что точно НЕ делаем в первой версии (out of scope)?

**Блок 3 — Архитектура:**
- Стек детально: frontend / backend / БД / хостинг (дефолт — из CLAUDE.md после bootstrap)
- Ключевые сущности данных и их связи?
- Внешние сервисы и интеграции (платежи, почта, AI-провайдеры…)?
- Модель аутентификации и ролей?

**Блок 4 — Операционка:**
- Команда сборки и тестов? (дефолт — `BUILD_COMMAND` из CLAUDE.md)
- Какие env-переменные понадобятся (имена, без значений)?
- Как деплоим?

## Phase 2 — Черновики

Показать в чате план: список файлов и краткое содержание каждого (3–5 буллетов на файл). `AskUserQuestion`: «Записать» / «Поправить (что именно)» / «Отмена».

## Phase 3 — Запись (после подтверждения)

| Файл | Содержание |
|---|---|
| `wiki/product/vision.md` | что строим, для кого, боль, ценность, отличие |
| `wiki/product/scope.md` | главный flow, второстепенные сценарии, in/out of scope |
| `wiki/architecture/system.md` | компоненты, стек, сущности данных, интеграции, auth |
| `CLAUDE.md` | заполнить секцию «Стек», команду сборки, стек-специфичные правила и запреты (вместо TODO-комментариев) |
| `PROJECT-ENTRYPOINT.md` | актуализировать Stack summary и стек-инварианты |
| `.env.example` | каркас секций с именами переменных (без значений) |

Правила записи:

- новые wiki-файлы — по структуре `wiki/_template.md`, с back-link `← [[index|Wiki]]`
- в `wiki/index.md` добавить ссылки на созданные документы
- открытые вопросы — отдельной секцией в конце каждого документа, где они остались
- ничего не удалять из существующих документов; CLAUDE.md и PROJECT-ENTRYPOINT.md правятся точечно

## Финал

Вывести список созданных/обновлённых файлов и напомнить: документы в статусе `generated` — после ревью человек переводит их в `draft`/`approved` вручную. Предложить (не выполнять автоматически): `/story-draft` для первой story, когда scope устаканится.
