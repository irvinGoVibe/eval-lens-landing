# Apple-block grammar — правила композиции блока

Это **единственный авторский контент скилла** (всё остальное — токены, рейлы,
каталог секций — подгружается живым из репо). Здесь — как должен быть устроен
блок в Apple-style, выраженный через **реальные токены и классы EvalLense**
(см. `web/src/app/globals.css`, `wiki/architecture/design-system.md`).

Apple-style — это **не визуальный пресет, а грамматика композиции**. Палитра и
типографика уже Apple (нейтральная база `--bg`/`--bg-soft`/`--ink` + lens-акценты).
Задача редизайна — навести **дисциплину ритма, иерархии и воздуха**, а не
добавить «красоты».

---

## 9 правил блока

### 1. Один блок — одна идея
Каждая секция делает **ровно одно утверждение**. Если в блок просится вторая
мысль — это второй блок. Лучше две чистые секции, чем одна перегруженная.
Тест: сформулируй идею блока одним предложением. Не получается — блок размыт.

### 2. Statement-first иерархия типа
Порядок и роли фиксированы (см. `.eyebrow` / `.title` / `.sub` в globals.css):
- `eyebrow` — мелкий маркер темы (caps/mono, `--muted`), 1–3 слова.
- **`heading` (`.title`)** — крупный display-заголовок (`--font-display`,
  плотный tracking, большой `clamp()`). **Несёт сообщение целиком.**
- `subhead` (`.sub`) — одна-две строки поддержки, не пересказ заголовка.
- `body` — опционально; если нужен абзац — возможно, идей две (см. правило 1).

Заголовок должен читаться **самостоятельно**, без подзаголовка. Подзаголовок
добавляет конкретику (число, механизм), а не повторяет.

### 3. Воздух — это и есть дизайн
Драматичные вертикальные отступы между блоками и вокруг контента. Не бойся
пустоты — «make it more Apple» исторически = «добавь воздуха». Используй
ритм секций из `page-design-patterns.md`; не ужимай ради «уместить больше».

### 4. Один герой-визуал на блок
Один крупный визуал (центр или full-bleed) = доказательство идеи блока. **Не**
сетка из мелких картинок там, где работает один большой кадр. Нет ассета —
видимый плейсхолдер `.media-ph` (лейбл + `--ratio`), а не пустота и не заглушка
«lorem».

### 5. Поверхность под замысел — и бесшовная стыковка
Осознанно назначь блоку поверхность: **light** (`--bg`/`--bg-soft`, текст `--fg`)
или **ink** (`--ink`/`--panel`, текст `--fg-on-dark`).

**Чередование НЕ обязательно.** Поверхность выбирается под замысел страницы, а не
по правилу «light↔ink». Страница может быть преимущественно светлой или
преимущественно тёмной; несколько light подряд или несколько ink подряд —
это нормально и часто желательно (цельный тон ровнее «кино»).

Настоящая работа здесь — **стыковка соседних блоков одной поверхности так, чтобы
шва не было видно**:
- Соседние одноцветные секции делят **одну базу фона** (один и тот же токен
  `--bg`/`--ink`) — не «почти такой же» оттенок, иначе виден стык.
- Граница между ними — это **переход, а не стена**: либо честный hairline-разделитель
  (`--border` / `--border-on-dark`), либо мягкое затухание/общий фон, либо
  продуманный вертикальный ритм отступов. Выбери одно и держи единообразно.
- Не клади жёсткую тень/резкую кромку между двумя light (или двумя ink) блоками —
  это читается как «сшито из кусков».
- Переход light→ink (если он есть) — самое заметное место: оформи его осознанно
  (контактная кромка, общий мост-элемент вроде `MistBridge`/`OrangeGlow`), а не
  резким обрывом.

Hero и финальный CTA могут быть любой поверхности — по тону страницы, не по
догме. Цель — **единый, хорошо стыкующийся холст**, а не обязательный контраст.

### 6. Грамматика движения — сдержанная и осмысленная
Движение **только** через data-атрибуты, оркеструемые в `ScrollOrchestrator.tsx`
(единый rAF-цикл — НИКОГДА не свой `useEffect` на анимацию):
- `data-reveal` (fade/up/left/right/scale) — появление при входе в вьюпорт. База.
- `data-scrub` — привязка прогресса к скроллу (последовательный reveal).
- `data-pin` / `data-pin-steps` — pinned multi-screen для «как это работает».

Easing — медленный Apple-кубик `--ease`. Никаких пружин, отскоков, «движения
ради движения». Один блок — один тип движения; не комбинируй три эффекта.

### 7. Сдержанность акцента
Нейтральная Apple-база **+ один** lens-акцент на блок (lens-градиент на одном
слове заголовка через `--lens`, или один цветной элемент). Акцент ведёт глаз к
**самому важному слову**. Цвет — семантичен (`--green` ready, `--amber`
attention), не декоративен. Два конкурирующих акцента убивают Apple-ощущение.

### 8. Контейнер и measure
Контент в `--maxw` с `--gutter`. Длина строки текста — читаемая (не во всю
ширину экрана). Заголовок может быть шире, body — уже. Центрирование —
по умолчанию для statement-блоков.

### 9. Сохрани контракт компонента
Редизайн **визуальный**, не функциональный: пропсы/API компонента, серверность
(Server Component для секций), его место в `ScrollOrchestrator` и данные —
сохраняются. Если редизайн просит новый пропс — делай блок **prop-driven** в
духе `sections/lab/` (различия = пропсы, не форки компонента).

## Анти-паттерны (что убивает Apple-ощущение)

- Несколько конкурирующих идей в одном блоке.
- Видимый шов между двумя одноцветными секциями: чуть разные оттенки фона,
  резкая кромка или жёсткая тень на стыке — читается как «сшито из кусков».
- Чередование light/ink ради контраста, ломающее задуманный цельный тон страницы.
- Тесные отступы, контент «впритык».
- Сетка мелких визуалов вместо одного крупного кадра.
- Декоративные градиенты/тени/свечения «для красоты» по всему блоку.
- Движение ради движения; >1 тип анимации на блок; пружины/отскоки.
- >1 акцентного цвета, борющегося за внимание.
- Стена body-текста (признак того, что идей несколько).
- Хардкод цветов/радиусов/теней вместо токенов.
- Свой `useEffect` на скролл/анимацию вместо `ScrollOrchestrator`.
- Разовый glass-look вместо общей liquid-glass группы.

## Чек перед сдачей блока

- [ ] Идея блока формулируется одним предложением.
- [ ] Заголовок (`.title`) читается без подзаголовка; иерархия eyebrow→heading→sub→body.
- [ ] Один герой-визуал (или осознанный `.media-ph`), не сетка мелких.
- [ ] Поверхность light/ink выбрана под замысел страницы; стык с соседними блоками
      бесшовный (общая база фона, единый тип перехода), а не «сшито из кусков».
- [ ] Движение — через `data-*` в `ScrollOrchestrator`, easing `--ease`, один тип.
- [ ] ≤1 lens-акцент; цвет семантичен; всё остальное — нейтральная база.
- [ ] Воздух щедрый; контент в `--maxw`/`--gutter`.
- [ ] Только токены (CSS-vars), без хардкода.
- [ ] Пропсы/серверность/данные компонента сохранены; различия = пропсы.
- [ ] glass (если есть) — через общую liquid-glass группу, не локально.
