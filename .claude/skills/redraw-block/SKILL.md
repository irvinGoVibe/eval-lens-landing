# Redraw Block — апгрейд блока до продуктового качества

Ты — фронтенд-дизайнер EvalLense. Берёшь **один существующий структурный блок**
(`web/src/components/**` — скелет: какие секции, порядок, слоты контента) и
**доводишь его вёрстку до современного продуктового качества в Apple-style**,
оставаясь строго внутри дизайн-системы и сохраняя поведение.

Рабочий диалог — **на русском**. UI-копия в коде — на английском (язык сайта),
если не попросили иначе.

> **Главный принцип:** **переиспользуем структуру — апгрейдим исполнение.**
> Структурный скелет блока (какие секции, порядок, слоты, что показываем) —
> это **вход**, его сохраняем. Меняется **качество вёрстки**: иерархия, ритм,
> воздух, типографика, состояния, движение, финиш — до уровня «сделал senior
> product-дизайнер + senior фронтендер».
>
> Редизайн **визуальный, не функциональный.** Пропсы, API, серверность, данные
> и место компонента в `ScrollOrchestrator` сохраняются. Различия
> страниц/контекстов выражаются **пропсами**, а не форком компонента (паттерн
> `sections/lab/`, память «section-types-as-components»).
>
> Две оси работы: **грамматика Apple-блока** (что это за блок — композиция) +
> **планка craft-quality** (насколько хорошо сделан — финиш). Обе обязательны.

> **Scope:** один компонент за вызов. Сборку страниц из готовых блоков делает
> **build-pages** — этот скилл его кормит, а не заменяет.

## Шаг 1. Определи цель и сними baseline компонента

**Вход — имя компонента** (`Hero`, `Problem`, `Workflow`, `Button`, …) или путь.
Прочитай его файл **целиком** и зафиксируй контракт, который НЕЛЬЗЯ ломать:

- **Структурный скелет (вход, сохраняем)** — какие секции/части внутри блока,
  их порядок, слоты контента, что показываем. Это переиспользуемая структура;
  редизайн её НЕ меняет (только по явной просьбе user'а). Апгрейдим исполнение
  внутри этого скелета.
- **Пропсы / API** — сигнатура, что компонент принимает и рендерит.
- **Серверность** — Server Component или `"use client"`? (Секции — обычно Server.)
  Редизайн не должен превращать Server-секцию в клиентскую без причины.
- **Данные** — откуда контент (пропсы, `lib/blog.ts`, статика).
- **Движение** — какие `data-reveal` / `data-scrub` / `data-pin` уже есть и как
  их читает `ScrollOrchestrator`. Сохрани крючки, по которым он находит блок.
- **glass** — использует ли `<Button variant="glass">` / `.btn-glass`. Если да —
  материал берётся из общей liquid-glass группы (см. Шаг 3), не локально.
- **Токены vs хардкод** — что уже на CSS-vars, что захардкожено (кандидат на чистку).

Сформулируй **идею блока одним предложением** (грамматика, правило 1). Если идей
несколько — отметь это: возможно, блок надо разбить (но это решение user'а).

## Шаг 2.5. (Опционально) Эскиз вариантов через baoyu-design

**Только для крупного редизайна**, где не очевидно «как блок вообще должен
выглядеть» и хочется поперебирать 2-3 направления **до** правки боевого TSX.
Для мелких craft-апгрейдов — пропусти, иди сразу к Шагу 3.

- Вызови скилл **baoyu-design** — он строит self-contained HTML-мокапы (быстро,
  не трогая `web/`). Поперебирай варианты раскладки/иерархии блока.
- **Свяжи мокап с дизайн-системой EvalLense**, иначе он будет «красивый, но не
  наш» → переделка. У baoyu для этого есть `use-design-system` (грузит DS как
  binding visual constraint). Если loadable DS-bundle ещё не заведён — лучше
  один раз втянуть токены `globals.css` + компоненты через `import-from-html`/
  `create-design-system`, тогда все будущие мокапы on-brand.
- Выбери направление → `handoff-to-claude-code` пакует интент → возвращаешься
  сюда (Шаг 3+) и **портируешь выбранное в боевой TSX по рейлам**.

> Граница ответственности: baoyu — это **look/layout**. Скролл/pin/scrub/
> liquid-glass/3D в HTML-мокапе не воспроизводятся один-в-один — их не «копируй
> как в HTML», а реализуй боевыми механизмами (`ScrollOrchestrator`, glass-группа)
> на Шаге 4. Мокап — это направление, не исходник кода.

## Шаг 3. Спроектируй апгрейд: грамматика + планка качества

Структурный скелет (Шаг 1) **остаётся**. В его рамках реши, как поднять
композицию (`apple-block-grammar.md`) и финиш (`craft-quality-bar.md`):

1. **Архетип** — какой тип из `section-types.md` лучше несёт идею (statement-hero,
   pinned-multi-screen, stat-band, bento, horizontal-gallery, risk-control, quiet-cta…).
   Бери существующий тип с его классом-примером, не изобретай near-дубль. Архетип
   должен **лечь на существующий скелет**, а не переломать его.
2. **Поверхность** — `light` или `ink` под замысел страницы (правило 5).
   Чередование НЕ обязательно: несколько светлых или несколько тёмных подряд —
   норма. Проверь **стык с соседними блоками**: одинаковая поверхность → общая
   база фона + единый тип перехода (hairline / мягкое затухание / общий фон),
   чтобы шва не было видно.
3. **Иерархия типа** — eyebrow → `.title` → `.sub` → body; заголовок несёт смысл
   сам (правило 2).
4. **Визуал** — один герой-кадр или осознанный `.media-ph` (правило 4).
5. **Движение** — один тип через `data-*`, easing `--ease` (правило 6). Анимация
   живёт в `ScrollOrchestrator`, не в новом `useEffect`.
6. **Акцент** — ≤1 lens-акцент, остальное нейтральная база (правило 7).
7. **Финиш (craft-quality-bar)** — заранее наметь, где поднять качество: spacing
   из единой шкалы, типографика (`clamp`/line-height/tracking/`text-wrap`),
   оптическое выравнивание, состояния `hover/focus-visible/active/disabled`,
   отзывчивость, нулевой CLS. Это и есть продуктовый апгрейд исполнения.

Прогони замысел против анти-паттернов грамматики, планки `craft-quality-bar.md`
и чек-листа «не плоско ли» из `page-design-patterns.md`. Кратко покажи план
user'у перед правкой кода, если редизайн заметный.

## Шаг 4. Реализуй, не нарушая рейлы

Точечные правки в файле компонента (+ при необходимости `globals.css`). Держи:

- **Структура — не трогаем.** Правим вёрстку/стили внутри существующего скелета;
  порядок секций, слоты и состав контента сохраняются (менять — только по
  явной просьбе user'а).
- **Доведи финиш по `craft-quality-bar.md`** — это главный предмет работы:
  spacing-ритм, типографические детали, выравнивание, все интерактивные
  состояния, отзывчивость, отсутствие CLS, семантичная разметка.
- **Токены только через CSS-vars** — никакого хардкода цветов/радиусов/теней.
  Используй существующие имена из globals.css (`--bg`/`--bg-soft`/`--ink`/`--fg`,
  `--violet`/`--cyan`/`--aqua`, `--r-*`, `--sh-*`, `--lens`, `--maxw`, `--gutter`).
- **liquid-glass — только общая группа.** Если блоку нужен glass-материал —
  добавь его селектор в общую multi-selector группу в `globals.css` (поиск
  `liquid glass` / `.btn-glass`), НЕ изобретай локальный glass-look (рейл CLAUDE.md).
- **Анимации — только в `ScrollOrchestrator.tsx`** (единый rAF). Новый эффект —
  это новый `data-*` + обработка в оркестраторе, не per-section `useEffect`.
- **Секции остаются Server Components**; `"use client"` — только если уже был или
  реально нужен (интерактив). Не превращай статику в клиент без причины.
- **Prop-driven** — если редизайн вводит вариант, делай его пропсом в духе
  `sections/lab/` (различия = пропсы), а не копией компонента.
- **pnpm only** — никаких `npm install` / `package-lock.json`.

## Шаг 6. Отчёт в чат + чек-лист

Короткий отчёт по-русски:
1. **Что апгрейднул** — какой архетип/поверхность/иерархию выбрал и **что именно
   поднял в качестве вёрстки** (spacing, типографика, состояния, отзывчивость).
2. **Что сохранил** — структурный скелет, пропсы, серверность, данные, крючки
   `ScrollOrchestrator`.
3. **Рейлы** — токены/glass/анимации/pnpm соблюдены; что чистил от хардкода.
4. **Сборка** — результат `pnpm build`.
5. **Открытые вопросы** — если блок просил разбиения/новых данных/ассета.

Чек-лист сдачи:
- [ ] **Структурный скелет сохранён** (порядок секций/слоты/контент); апгрейднут финиш.
- [ ] **Планка `craft-quality-bar.md` пройдена** (spacing-ритм, тип-детали,
      выравнивание, все состояния, отзывчивость, нет CLS, семантичная разметка).
- [ ] Идея блока — одно предложение; иерархия типа statement-first.
- [ ] Архетип из `section-types.md` (не near-дубль); поверхность light/ink под замысел страницы.
- [ ] Стык с соседними блоками бесшовный (общая база фона, единый тип перехода), а не контраст ради контраста.
- [ ] Один герой-визуал / осознанный `.media-ph`; воздух щедрый; контент в `--maxw`.
- [ ] Движение через `data-*` в `ScrollOrchestrator` (НЕ новый `useEffect`); easing `--ease`; один тип.
- [ ] ≤1 lens-акцент; цвет семантичен; только токены, без хардкода.
- [ ] glass — через общую liquid-glass группу, не локально.
- [ ] Контракт сохранён: пропсы/API/серверность/данные целы; вариативность = пропсы.
- [ ] `cd web && pnpm build` проходит.
- [ ] pnpm, без npm/package-lock; порт 3005 не менялся.
