# CMS

← [[index|Wiki]] · [[system|System]]

Что такое CMS в проекте и какие функции она выполняет: какой контент через неё
редактируется и какие операции над ним доступны. Технический подход (файловая
модель, слой данных, выборка, миграция) описан в [[blog|Blog (Newsroom) —
архитектура]] §CMS-слой — здесь не дублируется.

## Подход (коротко)

**Бэкенд — Supabase**: записи блога (статьи и репосты) — в Postgres, картинки и
видео — в Supabase Storage. Сайт остаётся фронтом на Vercel, который читает
данные из Supabase на сервере. Редактирование — через собственную admin-страницу
в приложении, закрытую паролем (см. «Доступ и авторизация»); ключи Supabase —
только server-side в `.env`. CMS правит только **контент**, не поведение
(вёрстка, анимации, 3D — в коде). Технические детали (схема, выборка, медиа,
миграция) — [[blog|архитектура блога]] §CMS-слой.

## Доступ и авторизация

Доступ к редактированию закрыт **одним паролем** — без аккаунтов, ролей и личных
кабинетов. Пароль хранится в **env** (например `CMS_PASSWORD`): только
server-side, файл `.env` в `.gitignore`, в клиентский бандл секрет не попадает
(по `CLAUDE.md` §Запреты).

- Гейт стоит перед admin-страницей. Публичная часть сайта (`/`, `/blog`, …) —
  read-only и пароля не требует.
- Проверка пароля — **только на сервере** (route handler / server action /
  middleware): введённое значение сравнивается с `CMS_PASSWORD`. Сам пароль
  никогда не уходит в клиент.
- После успешного входа — httpOnly-кука / подписанная сессия, чтобы не вводить
  пароль на каждое действие.
- Ротация — сменить значение в `.env` и в переменных окружения на Vercel.

Это сознательно простой гейт для маленькой внутренней CMS: один общий секрет, без
многопользовательской модели. Роли/несколько редакторов — отдельное решение на
будущее, если понадобится.

## Визуальный язык и интерфейс

CMS **следует дизайн-системе продукта** ([[design-system]]): светлая база
(`--bg`/`--bg-soft`), фиолетовый primary (`--violet`), SF Pro/Menlo,
hairline-границы, мягкие фиолетово-подкрашенные тени (`--sh-1/2/3`), канонические
радиусы (`--r-xs..pill`). Стили живут в `web/src/app/admin/admin.css` (scoped под
`.admin-root`) и берут только токены из `globals.css :root` — отдельной палитры у
админки больше нет.

**Интерфейс CMS — на английском** (навигация, лейблы форм, кнопки, состояния,
сообщения). i18n-инфраструктуры нет: строки захардкожены EN в admin-зоне (это
приемлемо для внутренней админки, не для публичной части). Значения данных
(категории Press Release / Product / Research, `status`) не переводятся.

> **Реализовано в Story 05.** `admin.css` переписан с тёмной утилитарной палитры
> на токены дизайн-системы (светлая тема, violet primary); весь интерфейс
> локализован на английский.

> **Реализовано в Story 02.** Админка закрыта паролем: проверка `CMS_PASSWORD`
> только на сервере (сравнение фиксированных SHA-256-дайджестов в constant-time),
> после входа — httpOnly-кука с дайджестом (пароль в куку не пишется).

## Что редактирует CMS

Три области контента.

### 1. Статьи Newsroom (`Post`) — уже есть в модели

Создание и ведение статей блога. Запись — строка таблицы `articles` в Supabase,
обложка и картинки тела — объекты Storage. Поля и блоки тела описаны в
[[blog|Blog (Newsroom)]] (`Post` / `Block`).

Через CMS редактор может:

- создать / изменить / удалить статью;
- задать рубрику (одна из трёх: Press Release / Product / Research), обложку,
  автора и роль, дату (определяет порядок), время чтения, короткое описание;
- собрать **тело** из блоков — текстовых (абзац, подзаголовок, цитата, список) и
  медиа (картинка, слайдер/галерея, видео);
- держать статью черновиком и опубликовать.

#### Создание тела статьи: Markdown-редактор

Тело статьи пишется в **Markdown-редакторе**. Редактор не собирает блоки руками —
он пишет Markdown, а блоки (заголовки, абзацы, цитаты, списки, картинки)
получаются из разметки. Исходный Markdown хранится в записи статьи; на рендере
он превращается в те же стили, что и сейчас.

> **Реализовано в Story 02.** Markdown-этап готов: тело хранится как text,
> рендерится remark-рендерером с маппингом на `article-*` стили и директивами
> `::video` / `:::gallery`.

**Поверхность редактирования (Story 05.1).** Markdown остаётся единственным
каноном тела (сохраняется verbatim-строкой), но поверхность редактирования
доведена до продуктового инструмента поверх обычной `<textarea>`
(прогрессивное улучшение — без JS поле остаётся редактируемым):

- **Тулбар форматирования** — ghost icon-кнопки (bold/italic/inline-code,
  H2/H3, quote, bullet/ordered list, link, divider). Действия применяются к
  выделению/курсору чистыми утилитами (`wrapSelection` / `prefixLines` /
  `insertAtCursor`), восстанавливая выделение. Хоткеи: Cmd/Ctrl-B, -I, -K.
- **Вставка медиа по курсору** — drag-and-drop файла прямо в текст, paste из
  буфера и кнопка выбора файла вставляют `![](url)` / `::video{src=}` **в
  позицию каретки** (а не в конец тела). Во время загрузки на месте стоит
  уникальный inline-плейсхолдер; по успеху он заменяется на финальный snippet
  по токену (устойчиво к правкам во время загрузки), по ошибке — удаляется с
  понятным сообщением. Upload по-прежнему идёт **через server action**
  `uploadArticleMediaAction` (секреты в клиент не попадают).
- **Слэш-меню `/`** — в начале строки/после пробела открывает
  клавиатурно-навигируемое меню блоков и директив (Heading H2/H3, Quote,
  Bullet/Ordered list, Image=выбор файла, Video=`::video`, Gallery=`:::gallery`,
  Divider) — чтобы директивы были обнаружимы без знания синтаксиса. Esc
  закрывает; в середине текста не триггерится.
- **Табы Write / Preview (+ Split)** — сегментед-контрол вместо тесного 50/50;
  на широком экране (≥ ~1100px) доступен Split с пропорциональным синхро-скроллом
  (гасится при `prefers-reduced-motion`). Превью остаётся на белом `.article-*`
  canvas, не наследуя admin-тему.
- **Автосейв черновика** — тело автосейвится в localStorage (debounce ~800 мс,
  ключ `evallens:draft:article:<slug|new>`); при расхождении локального
  черновика с серверным телом сверху редактора показывается ненавязчивый баннер
  «Found an unsaved draft — Restore / Discard» (по умолчанию показано серверное
  тело). Уход с несохранёнными изменениями предупреждает; после успешного Save
  черновик очищается.
- **Футер-мета** — счётчики слов/символов и оценка времени чтения (информативно;
  `read_minutes` остаётся ручным полем).

> **Реализовано в Story 05.1.** Канон (Markdown), публичный рендер
> (`MarkdownBody`), директивы и схема данных — без изменений; обогащена только
> поверхность редактирования.

Какие конструкции Markdown поддерживаем и во что они рендерятся:

| Markdown | Элемент на странице |
| --- | --- |
| абзац (текст через пустую строку) | `article-p` |
| `## Заголовок` | `article-h2` |
| `> цитата` | `article-quote` |
| `- пункт` / `1. пункт` | `article-list` |
| `![подпись](url-в-Storage)` | картинка статьи |
| `::video{src=… poster=…}` | видео в теле |
| `:::gallery … :::` | галерея / слайдер |

**Галерея и видео — через Markdown-директивы** (`remark-directive`). MDX не
берём: тело хранится в Supabase как текст и рендерится в рантайме, поэтому оно
должно оставаться **данными** (декларативная разметка), а не исполняемым кодом.
Синтаксис:

- **Видео** — leaf-директива (`poster` опционален):

  ```
  ::video{src="<storage-url>" poster="<storage-url>"}
  ```

- **Галерея / слайдер** — контейнер-директива с обычными картинками внутри:

  ```
  :::gallery
  ![](<storage-url-1>)
  ![](<storage-url-2>)
  ![](<storage-url-3>)
  :::
  ```

Директивы рендерятся своими компонентами: видео — тег `<video>` (как в ленте
«In the Loop»), галерея — клиентский листаемый компонент (логика из
`PopupGallery`).

- **Безопасность:** рендерим только поддерживаемый Markdown + наши директивы,
  **сырой HTML отключаем** — произвольная разметка и скрипты в контент не попадают.
- Картинки и медиа ссылаются на объекты **Supabase Storage** (URL вставляется
  при загрузке файла через CMS).

#### Загрузка медиа: drag-and-drop + превью

Загрузка изображений/видео в формах статей — **нативная HTML5 drag-and-drop зона**
(без сторонних пакетов): перетаскивание или выбор файла, локальное превью
(`URL.createObjectURL`), indeterminate-индикатор прогресса вокруг server-action
(побайтовый стрим из server-action недоступен — это ожидаемо) и понятные ошибки
(неподдерживаемый тип, превышение размера, сбой). Остаётся фолбэк: обычный
`<input type=file>` внутри зоны и ручной ввод URL. Сам upload идёт **через server
action** (`uploadArticleMediaAction` → `uploadMedia`), секреты в клиент не
попадают. Применяется к обложке статьи и к вставке изображений/видео в тело
(добавляет `![](url)` / `::video{src=}`).

> **Реализовано в Story 05.**

### 2. Репосты In the Loop (`LoopPost`) — уже есть в модели

Создание и ведение ленты репостов. Запись — строка таблицы `loop_posts`, видео
и фото — объекты Storage. Поля описаны в [[blog|Blog (Newsroom)]] (`LoopPost`).
Своей страницы у репоста нет — открывается попапом, есть deep-link `?loop=<id>`.

Через CMS редактор может:

- создать / изменить / удалить репост;
- выбрать формат (видео или фото/галерея) и приложить медиа (ролик / набор
  фото);
- задать подпись, автора и инициалы, дату, ссылку на оригинал в соцсети;
- влиять на порядок в ленте.

> **Реализовано в Story 05.** Форма репоста умеет грузить файлы для cover, video
> и галереи через server action `uploadLoopMediaAction` → `uploadMedia(file,
> 'bento'|'video'|'photos')` (та же drag-and-drop зона, что и в статьях). Ручной
> ввод URL остаётся как фолбэк. Раньше форма принимала только URL — разрыв «cms.md
> упоминает медиа репостов в Storage, а форма этого не умеет» закрыт.

### 3. Блок с блогом на главной — единственное динамическое на главной

На главной странице через CMS управляется **только блок с блогом** (лента
новостей/статей, которая тянет записи из Supabase). **Всё остальное на
главной — статика:** секции Hero, Problem, OrangeGlow, Workflow, Decisions, Bento
(`web/src/components/sections/`) остаются захардкоженными в коде и через CMS не
редактируются.

В админке блок блога на главной — отдельный раздел в папке **Главная**. Через
него редактор:

- **выбирает, какие статьи** показываются в блоке на главной, и **задаёт их
  порядок** (featured-выборка + порядок);
- в выборку попадают только `published`-статьи; при пустой выборке — фолбэк на
  свежие по `date` (как было до выборки).

Редактируется именно **выбор и порядок статей**, а не текст/вёрстка самой секции
(заголовок, оформление блока — в коде). Отдельной контент-сущности под блок не
заводим: он показывает те же статьи, что и раздел `/blog`. Остальные секции
правятся только разработчиком в коде — на старте незачем выносить весь лендинг
в CMS.

> Изменение направления: ранее тут планировалось делать редактируемым контент
> всех секций главной — **отменено**. На главной из CMS редактируется только
> блок блога (выбор и порядок статей).

> **Реализовано в Story 02.** Главная теперь редактируема в этой части: редактор
> управляет выбором и порядком статей в блоке «Блог» на главной (раздел «Главная»
> в админке). Остальные секции главной (Hero / Problem / … ) через CMS не
> редактируются.

## Функции CMS (сводно)

| Операция | Статьи | Репосты | Главная (блок «Блог») |
| --- | --- | --- | --- |
| Создать / удалить | да | да | нет — секции статичны |
| Редактировать | да | да | выбор и порядок показываемых статей |
| Медиа (загрузка в Storage) | обложка + медиа в теле | видео / галерея | — |
| Порядок | по `date` | порядок в ленте | задаётся редактором (фолбэк — по `date`) |
| Черновик / публикация | да (`status`) | да (`status`) | показываются только `published` |

## Что CMS НЕ делает

- не управляет вёрсткой, анимациями и 3D-сценой — это код;
- не редактирует статичные секции главной (Hero / Problem / … ) — на главной из
  CMS управляется только блок блога (выбор и порядок статей), не его вёрстка;
- не заводит страницы под репосты (только лента + попап + deep-link);
- не хранит пользовательских данных и не даёт публичной записи (read-only фронт;
  запись — только через Supabase server-side).

## Открытые вопросы

- Проект Supabase ещё не создан; схему и Storage нужно завести и залить контент.
- UX публикации в админке (`status: draft → published`) — детали не
  детализированы (само решение «один пароль, без ролей» зафиксировано).
- **Осиротевшие объекты Storage.** `uploadMedia` использует `upsert:false` и имя
  `<safe-name>-<Date.now()>.<ext>`, поэтому перезаливка cover/медиа создаёт новый
  объект, а старый остаётся в бакете. Чистка/удаление объектов Storage из UI —
  вне scope Story 05 (медиа-библиотека / браузер бакета не реализованы);
  поведение зафиксировано, не решается здесь.
