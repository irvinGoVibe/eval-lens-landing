# Blog (Newsroom)

← [[index|Wiki]] · [[system|System]]

Как устроен раздел `/blog` (EvalLense Newsroom): какие сущности контента
существуют, какие у них параметры, как из них собираются страницы и что такое
«In the Loop» репосты.

> **Текущее состояние vs план.** Сейчас источник правды — статический файл
> `web/src/lib/blog.ts` (массивы прямо в коде, добавление = коммит в git).
> **Целевое состояние — Supabase:** контент хранится в Supabase — записи статей и
> loop-постов в Postgres, картинки/видео в Supabase Storage; редактируется через
> Supabase, а не правкой кода. Доменная модель ниже (`Post`, `LoopPost`, `Block`)
> сохраняется — меняется источник данных: вместо массивов из `blog.ts` хелперы
> читают из Supabase. Подробно — раздел «CMS-слой».

## Информационная архитектура

Newsroom повторяет структуру Apple Newsroom тремя блоками:

- **Latest News** — featured-герой + боковой rail из 3 свежих статей.
- **In the Loop** — горизонтальный rail репостов соцсетей (видео + фото),
  открываются в попапе.
- **More from Newsroom** — четыре статичные плитки-категории.

Маршруты:

- `/blog` — хаб: Latest News + In the Loop + More from Newsroom.
- `/blog/all` — полный список статей (грид с пагинацией «Show more»).
- `/blog/[slug]` — страница статьи + блок «More from the Newsroom».

**Только у статьи (`Post`) есть свой маршрут (`/blog/[slug]`).** У loop-поста
отдельной страницы нет: он живёт в ленте «In the Loop» и открывается в попапе
прямо на `/blog`. Но поддерживается **deep-link** — `/blog?loop=<id>` открывает
попап нужного репоста сразу при загрузке (ключ — `LoopPost.id`). Это даёт
расшариваемую ссылку на конкретный репост внутри сайта, не заводя под него
отдельную страницу. Кнопка «Share ↗» в попапе при этом по-прежнему ведёт на
оригинал в соцсети (`href`).

Следствия:

- `generateStaticParams` строится **только по статьям**; loop-посты тянутся как
  данные для ленты, отдельные страницы не генерируются.
- `InTheLoop` (клиентский) читает `?loop=<id>` через `useSearchParams` и
  открывает соответствующий попап на маунте; при открытии / закрытии /
  перелистывании синхронизирует URL (`router.replace`, без скролла), чтобы
  прямая ссылка и кнопка «назад» работали корректно.

Весь контент-слой — `web/src/lib/blog.ts`. UI — Server Components в
`web/src/components/blog/`, кроме двух клиентских (`InTheLoop`, `AllNewsGrid`).
Стили — legacy-CSS в `web/src/app/globals.css` (классы `blog-*`, `loop-*`,
`article-*`).

## Сущности

По факту в контенте **две сущности**:

1. **`Post` — статья** (раздел Latest News / All News, страница `/blog/[slug]`).
2. **`LoopPost` — репост «In the Loop»** (горизонтальный rail с попапом).

Всё остальное — не самостоятельные сущности, а вспомогательные части:

- **`Block`** — строительный блок *тела статьи*: статья хранит текст не одной
  строкой и не как HTML, а как массив типизированных блоков (`Post.body:
  Block[]`). Существует только внутри `Post`, отдельно не живёт. Зачем так —
  см. ниже в разделе про `Block`.
- **`Category`, `Accent`, `LoopKind`** — фиксированные наборы значений (enum-ы),
  поля внутри сущностей, а не отдельные записи.

### `Post` — статья

Полноценная редакционная статья. Массив `POSTS` в `blog.ts`; добавление статьи =
новая запись в массиве, страницы подхватывают её автоматически (новые сверху по
`date`).

| Поле | Тип | Назначение |
| --- | --- | --- |
| `slug` | `string` | URL-идентификатор; формирует `/blog/{slug}`. Должен быть уникальным. |
| `category` | `Category` | Рубрика; показывается тегом (`CategoryTag`). |
| `accent` | `Accent` | Акцентный цвет тега и hover-эффектов карточки. |
| `title` | `string` | Заголовок статьи (карточка + страница). |
| `excerpt` | `string` | Одно предложение-dek: на карточках и в шапке статьи. |
| `date` | `string` | ISO-дата (`YYYY-MM-DD`). Управляет сортировкой и человекочитаемым штампом. |
| `readMinutes` | `number` | Оценка времени чтения («N min read»). |
| `cover` | `string` | Путь к обложке под `/public` (напр. `/assets/bento/jury-decision.png`). |
| `author` | `string` | Автор / источник (напр. «EvalLense Newsroom»). |
| `role` | `string` | Подпись роли под автором (Company / Design / Security…). |
| `body` | `Block[]` | Тело статьи — массив блоков (см. ниже). |

### `Block` — блок тела статьи (часть `Post`, не отдельная сущность)

**Зачем он нужен.** Тело статьи могло бы храниться как одна строка HTML или
markdown — но тогда вёрстка статьи зависела бы от того, что наредактировали в
тексте, и пришлось бы парсить/санитайзить произвольную разметку. Вместо этого
тело — это *структурированный* массив блоков: каждый элемент `body` явно говорит,
что он такое (`p` — абзац, `h2` — подзаголовок, `quote` — цитата, `list` —
список). Преимущества:

- единый контроль вёрстки — каждый тип блока рендерится своим компонентом и
  своим CSS-классом (`article-p`, `article-h2`, `article-quote`, `article-list`);
- никакого сырого HTML в контенте → нет XSS и «расползания» стилей;
- легко расширять (добавить `image`/`embed` — это новый вариант `type`) и легко
  лечь в rich-text поле будущей CMS.

Технически это дискриминированное объединение по полю `type`; рендерится в
`app/blog/[slug]/page.tsx` (`BlockContent`).

| `type` | Поля | Рендер |
| --- | --- | --- |
| `p` | `text` | абзац `<p class="article-p">` (значение по умолчанию) |
| `h2` | `text` | подзаголовок `<h2 class="article-h2">` |
| `quote` | `text`, `cite?` | `<blockquote>` с опциональной подписью `<cite>` |
| `list` | `items: string[]` | маркированный `<ul class="article-list">` |

Других типов блоков нет — markdown/HTML внутри `text` не парсится, это plain text.

**Планируется** (в коде ещё нет): медиа-блоки `image` (картинка с подписью),
`gallery`/`slider` (листаемая галерея, как в «In the Loop») и `video`
(встроенный ролик). Это требование к контенту из продукта ([[newsroom|Newsroom]]).

> **Авторинг под CMS — Markdown.** Текущий код хранит тело как структурный
> `Block[]`; при переезде в Supabase тело **пишется и хранится как Markdown**, а
> рендерится Markdown-рендерером с маппингом на те же стили (`article-p` и т.д.).
> Текстовые блоки покрываются стандартным Markdown; галерея/видео — Markdown-
> директивами (`remark-directive`: `:::gallery`, `::video`). Подробно — §CMS-слой
> и [[cms|CMS]].

### `Category` и `Accent`

`Category` — фиксированный набор рубрик. **В коде сейчас 6** значений:
`"Press Release" | "Update" | "Quick Read" | "Feature" | "Research" | "Photos"`.

**Решено сократить до трёх** (минимально достаточный набор на старте, см.
[[newsroom|Newsroom]]): `"Press Release" | "Product" | "Research"`. Приведение
типа `Category`, перекатегоризация постов и плиток `MoreFromNewsroom` —
отдельная dev-задача; до неё в коде остаётся 6 значений (расхождение
зафиксировано осознанно).

`Accent` — акцентная тема (тег + hover): `"violet" | "cyan" | "aqua" | "orange"`.
Прокидывается в CSS через `data-accent` на карточках, тегах и плитках.

### `LoopPost` — репост «In the Loop»

Репост из соцсетей в rail «In the Loop». Массив `LOOP_POSTS` в `blog.ts`.
Два формата задаются полем `kind`.

| Поле | Тип | Назначение |
| --- | --- | --- |
| `id` | `string` | Уникальный ключ (React key + счётчик в попапе). |
| `kind` | `LoopKind` | `"video"` или `"photo"` — определяет формат карточки и попапа. |
| `author` | `string` | Имя автора репоста. |
| `initials` | `string` | Инициалы в градиентном аватаре (напр. «EL»). |
| `accent` | `Accent` | Акцент аватара и карточки. |
| `caption` | `string` | Подпись; в попапе фото первое предложение выделяется как lead. |
| `cover` | `string` | Постер-кадр (video) или главное изображение (photo); путь под `/public`. |
| `video?` | `string` | Путь к клипу под `/public`. **Обязателен** для `kind: "video"`. |
| `photos?` | `string[]` | Галерея для `kind: "photo"` (включая обложку). 2+ элементов → галерея с навигацией. |
| `date` | `string` | ISO-дата; показывается как «June 8, 2026». |
| `href` | `string` | Ссылка на оригинальный пост в соцсети (кнопка «Share ↗»). |

Поведение по `kind`:

- **`video`** — тёмная карточка; клип авто-играет при наведении (muted, loop), в
  попапе — плеер с `controls`/`autoPlay`. Нужно поле `video`.
- **`photo`** — светлая карточка с изображением сверху и текстом снизу,
  zoom при наведении. В попапе — карусель по `photos` (если 2+); если `photos`
  не задан, используется `[cover]`. Глиф-стопка показывается только при галерее.

## Хелперы доступа к данным (`blog.ts`)

| Функция | Возвращает | Где используется |
| --- | --- | --- |
| `getAllPosts()` | `Post[]`, отсортированный по `date` (новые сверху) | хаб, `/blog/all`, `generateStaticParams` |
| `getPostBySlug(slug)` | `Post \| undefined` | страница статьи, `generateMetadata` |
| `getRelatedPosts(slug, limit = 3)` | до `limit` других статей | блок «More» внизу статьи |
| `formatDate(iso)` | строка вида «June 10, 2026» (UTC, en-US) | карточки, шапка статьи, loop-карточки |
| `getLoopPosts()` | `LoopPost[]` (в порядке массива, без сортировки) | хаб (`InTheLoop`) |

Сортировка статей — стабильная по `date` (`getTime()` desc) и вычисляется один
раз на модуле. `LoopPost` не сортируется — порядок rail = порядок массива.

## Как добавить контент

**Сейчас (статический git-флоу):**

- **Статья** — добавить запись `Post` в массив `POSTS` (`blog.ts`). `slug`
  уникален, `cover` — существующий файл под `web/public`, `body` — массив
  `Block`. Сортировка и страницы — автоматически.
- **Репост In the Loop** — добавить `LoopPost` в `LOOP_POSTS`. Для `video`
  обязательно поле `video`; для галереи — `photos` с 2+ путями. Медиа — под
  `web/public`.

**План:** контент переезжает в Supabase (Postgres — записи, Storage — медиа).
Хелперы (`getAllPosts` и т.д.) станут адаптером над `supabase-js`, доменная
модель сохранится. Подробно — раздел «CMS-слой».

## CMS-слой (целевая архитектура)

Раздел описывает, как блог работает после переезда контента в Supabase. Уровень —
средний (для разработчика): что где живёт, какие контракты сохраняются и какие
решения ещё открыты. **Направление выбрано — Supabase** (Postgres + Storage).

Ключевая идея: разделяем **что хранится** (контент в Supabase) и **как
показывается** (тот же фронт). Доменная модель (`Post`, `LoopPost`, `Block`) и
компоненты не меняются — меняется только источник данных.

### Принцип: CMS прячется за адаптером

Контракт хелперов (`getAllPosts`, `getPostBySlug`, `getRelatedPosts`,
`getLoopPosts`, `formatDate`) **остаётся прежним**. Сегодня их реализация читает
массивы из `blog.ts`; цель — те же сигнатуры, но внутри запрос к CMS + маппинг
ответа в доменные типы. Тогда страницы (`app/blog/**`) и компоненты не трогаются
вообще — переписывается один слой данных.

Рекомендуемая раскладка вместо одного `lib/blog.ts`:

- `lib/cms/client.ts` — клиент Supabase (`supabase-js`); ключи из env, только
  server-side;
- `lib/cms/map.ts` — маппинг строк БД → доменные `Post` / `LoopPost` / `Block`
  (нормализация дат, ссылок на Storage, JSONB-тела);
- `lib/blog.ts` — публичные хелперы с прежними сигнатурами поверх двух модулей.

Так смена бэкенда или возврат к статике затрагивает только `lib/cms/*`.

### Где живёт контент: Supabase (Postgres + Storage)

Выбранное направление — **Supabase как бэкенд блога**:

- **Postgres** — записи: таблицы `articles` (статьи) и `loop_posts` (репосты).
- **Supabase Storage** — медиа: обложки и картинки статей, постеры и видео
  (бакеты с публичным read-доступом; при необходимости — подписанные URL).

Контент перестаёт быть кодом в репозитории — он живёт в Supabase. Сайт остаётся
фронтом на Vercel, который читает данные из Supabase. Это headless-DB подход
(не файлы, не git-CMS).

**Редактирование:** записи и загрузка картинок — через Supabase (на старте —
Supabase Studio / table editor; при необходимости позже — отдельная админка).
Конкретная поверхность редактирования — открытый вопрос.

**Доступ и секреты (важно, см. `CLAUDE.md` §Запреты):** ключи Supabase — только
в `.env`, server-side; в клиентский бандл не попадают. Публичный контент читаем
**на сервере** (Server Components) через `supabase-js`; прямых запросов из
браузера к таблицам нет. Публичный read — **anon-ключ + RLS** (Row Level
Security) с политикой «только `published`»; **service-ключ** — только для записи
из админки (server-side).

**Что меняется в коде:** слой данных `lib/cms/` (клиент Supabase + маппинг строк
БД в `Post` / `LoopPost` / `Block`) заменяет чтение массивов. Публичные хелперы
(`getAllPosts` и т.д.) — **те же сигнатуры**, фронт не трогаем.

### Контент-модель (таблицы Supabase)

Две таблицы, повторяющие сущности (поля — из таблиц `Post` / `LoopPost` выше):

- **`articles`** ← `Post`: `slug` (text, unique), `title`, `excerpt`, `category`
  (enum/CHECK из 3 рубрик), `cover` (text — путь/URL объекта в Storage),
  `author`, `role`, `date` (timestamptz), `read_minutes` (int), `body` (jsonb —
  массив блоков), `accent` (опц.) + служебные `id`, `status` (`draft`/`published`),
  `created_at` / `updated_at`.
- **`loop_posts`** ← `LoopPost`: `id`/`slug` (text — ключ deep-link `?loop=<id>`),
  `kind` (enum video/photo), `author`, `initials`, `caption`, `cover`, `video`
  (опц.), `photos` (опц. — `text[]`/jsonb), `date` (timestamptz), `href`,
  `status`, `created_at`.

**Тело статьи (`body`)** — храним как **Markdown-исходник** (text-поле): редактор
пишет тело в Markdown-редакторе (см. [[cms|CMS]]). Поддерживаемые конструкции
Markdown соответствуют нашим блокам: абзац → `p`, `##` → `h2`, `>` → `quote`,
`-`/`1.` → `list`, `![](…)` → картинка. Галерея/слайдер и видео — через
**Markdown-директивы** (`remark-directive`): `::video{src=… poster=…}` и
контейнер `:::gallery … :::` с картинками внутри (синтаксис и рендер — [[cms|CMS]]).
MDX не берём — тело должно оставаться данными, а не исполняемым кодом. Сырой HTML
в Markdown **отключаем** (нет XSS). Картинки/видео ссылаются на объекты Storage.

Рубрика — 3 значения (Postgres enum или `CHECK`-ограничение).

### Выборка данных (Next.js 16, Cache Components)

Контент в Supabase читаем **на сервере** (Server Components) через `supabase-js` —
никаких запросов из браузера. В клиент уходит только готовый HTML; `InTheLoop` /
`AllNewsGrid` остаются клиентскими, но получают данные пропсами, как сейчас.

Чтобы блог был быстрым (как статика) и при этом свежим — Cache Components
(включается `cacheComponents: true` в `next.config.ts`):

- хелперы-выборки помечаем `'use cache'` и тегируем `cacheTag('blog')` +
  `cacheTag('post:'+slug)`; срок — `cacheLife('days')` или с явным `revalidate`;
- `generateStaticParams` на `/blog/[slug]` берёт slug'и из Supabase на сборке;
- обновление контента → **Database Webhook** Supabase дёргает route handler,
  который делает `revalidateTag('blog')` / `revalidateTag('post:'+slug)` —
  свежесть без ре-деплоя.

> Версия Next нестандартная (см. `web/AGENTS.md`) — конкретные сигнатуры
> (`use cache`, `cacheLife`, `cacheTag`, `revalidateTag`) перед реализацией
> сверить с `node_modules/next/dist/docs/`.

### Медиа (обложки, видео, галереи)

Сейчас файлы лежат под `/public` и идут через `next/image` (обложки) и `<video>`
(клипы). С Supabase медиа переезжает в **Supabase Storage**:

- изображения — `next/image` + `images.remotePatterns` в `next.config.ts` под
  домен Supabase Storage; `sizes` в компонентах уже заданы, менять не нужно;
- видео — публичный URL объекта Storage в теге `<video>`;
- галереи — массив URL объектов Storage.

Загрузка медиа — через админку. Бакеты — **публичные** (контент на сайте и так
публичный); подписанные URL не используем.

### Рендеринг и блоки

Карточки и страницы — те же Server Components. Меняется рендер **тела статьи**:
вместо нынешнего `BlockContent` (switch по `type` структурного `Block[]`) —
**Markdown-рендерер** с маппингом на наши стили: `h2 → article-h2`,
`p → article-p`, `blockquote → article-quote`, списки `→ article-list`,
картинка `→` компонент изображения. Сырой HTML отключён. Директивы
`remark-directive` рендерятся своими компонентами: `::video` → `<video>`,
`:::gallery` → клиентский листаемый компонент (логику листания переиспользуем из
`PopupGallery` в `InTheLoop`).

Тег рубрики и плитки `MoreFromNewsroom` — привязать к 3 рубрикам (сейчас лейблы
плиток устарели и не фильтруют — см. открытые вопросы).

## Открытые вопросы

- **Бэкенд — Supabase** (решено): Postgres + Storage; чтение — anon-ключ + RLS
  «только `published`», запись — service-ключом из админки; медиа — публичные
  бакеты; видео — в Storage; редактирование — admin-страница под паролем
  ([[cms|CMS]]). Осталось как **задача миграции**: проект Supabase ещё не создан,
  схема/Storage не заведены, контент не залит. До миграции источник правды —
  `blog.ts`.
- `MoreFromNewsroom`: фильтрацию по рубрике на старте решено **не делать** —
  плитки остаются ссылками на полный список `/blog/all` (лейблы выровнять с 3
  рубриками — косметика в стори про рубрики).
- `cover` у статей не уникальны (несколько постов делят одну картинку) — это
  осознанно для демо-контента, но при реальном наполнении потребует ревизии.
