# Page Build Skills — скиллы сборки и правки страницы

← [[index|Wiki]]

Карта скиллов, которые **собирают и правят внутренние страницы** EvalLense из
готовой DS-библиотеки: чем они отличаются, в каком порядке применяются и что
делает каждый. Фокус документа — **трио точечного редактирования**, добавленное
2026-06-26 (`zone-background`, `ds-ify-page`, `retype-section`): операции над
**уже существующей** страницей, по одной правке за вызов, с одним user-гейтом.

Источник истины — фактические `SKILL.md` в `.claude/skills/`. Документ generated,
сверять с кодом при расхождении. Общая карта всей автоматизации —
[[agents-orchestrators|Agents & Orchestrators]].

---

## 0. Два слоя сборки страницы

| Слой | Что делает | Скиллы |
|---|---|---|
| **A — оркестраторы сборки** | Собирают/перебирают страницу целиком из библиотеки: структура → раскраска → переходы; или batch | `page-skeleton` → `page-skin` → `page-motion` · `page-composer` · `build-pages` · `page-art-director` |
| **B — точечная правка** | Одна направленная операция над готовой страницей, без полной пересборки | **`zone-background`** · **`ds-ify-page`** · **`retype-section`** · `redraw-block` |

Граница: слой **A** строит страницу с нуля или глубоко реконструирует; слой **B**
хирургически меняет один аспект готовой страницы и останавливается. Трио ниже —
ядро слоя B.

**Общие инварианты обоих слоёв** (по факту кода):
- Контейнер страницы — `<main className="<page> section-lab ds">`. `section-lab`
  обязателен (иначе `.lab-*` стили схлопываются); `ds` — светлый DS-язык.
- Движок — `<ScrollFX/>` в конце `<main>` (видимость DS-компонентов завязана на
  него: без движка `data-reveal` остаётся `opacity:0`). Главная (`/`) — исключение
  (`ScrollOrchestrator`), её эти скиллы не трогают.
- Публичный API секций — **только барель** `@/components/ds`. Сырой `Lab*` без
  TODO в финальном коде не оставляем (канон CL-007).
- `surface` секции = **только тон** (`light|ink`), не фон.
- Не коммитят/не пушат/не трогают `.env`; dev-сервер сам не поднимают;
  `pnpm build` — только с разрешения user; pnpm, порт 3005.

## 1. `zone-background` — общий фон под группу секций

**Триггер:** `/zone-background` · «сделай один фон под эти секции» · «объедини
секции N–M под общий бэкграунд» · «сквозной фон light→dark под блок» ·
`zone-bg <route> <N–M> <light|dark|flip>`.

**Задача.** Объединить **идущие подряд** секции одной страницы под **один общий
фон** — обернуть их в **зону** (`.ds-zone`) с единым фоновым слоем, который
читается непрерывно за всей группой.

**Ключевая модель.** Фон — это **НЕ проп секции**, а отдельный слой `.ds-zone__bg`
под группой. Зона делает все вложенные `.band` прозрачными; секция через `surface`
задаёт **только тон** (light/ink), не фон. Поэтому «один фон под пятью секциями» =
одна зона + один bg-слот, а не пять одинаковых пропов.

**Три пресета тона** (готовые классы в `web/src/components/ds/ds.css`, искать
`.ds-zone`):

| Пресет | Когда | bg-слот | surface секций |
|---|---|---|---|
| **light** | тёплый градиент с переливами-блобами | `ds-zone__bg ds-canvas__bg--lobes` | `light` |
| **dark** | «вспышки» на чёрном | `ds-zone__bg ds-canvas__bg--lobes-dark ds-zone__bg--on` + 3 `__spark` | `ink` |
| **flip** | сквозной фон light→dark по скроллу | два fixed-слоя `--flip-light` + `--flip-dark is-dark` + `[data-pin]`-драйвер на стыке | до стыка `light`, после `ink` |

> Технический нюанс: `--lobes-dark` по умолчанию `opacity:0` (он спроектирован
> *проявляться* в flip-шве). Для статической тёмной зоны добавляется модификатор
> **`ds-zone__bg--on`** (добавлен в `ds.css` 2026-06-26 как чистый target скилла).
> Блобы (`CanvasBlobs`) и спарки — опциональные энхансеры, базовый градиент
> работает без них.

**Что выводит наружу.** Секции со своим full-bleed фоном, который зона **не**
подавляет: **Hero** (`lab-hero__bg`), **Cinema** (`ds-cinema`/видео), **Footer**
(`variant="dark"`) — остаются вне зоны (full-bleed акценты, они рвут зону).
HubMap / Bento / Gallery — **можно** внутрь: их balls-glass бэкдроп зона гасит
автоматически.

**Гейт.** План зоны (что внутри / снаружи / целевой `surface` каждой секции +
выбранный пресет) **до** перезаписи.

**НЕ делает:** не редизайнит секции (`redraw-block`), не делает переходы между
зонами (`page-motion`), не куёт новые фоновые пресеты (`visual-layer-forge`).
Контент/порядок секций — инвариант. Одна зона за вызов.

**Референс живьём:** `web/src/app/dev/canvas-bg/page.tsx` (все три случая: light-
группа → tone-flip → dark). Модель формализована в [[component-library|Visual
layers]] (тональные зоны).

## 2. `ds-ify-page` — перевод страницы на канонические DS-компоненты

**Триггер:** `/ds-ify-page` · «переведи страницу `<route>` на DS» · «DS-ифицируй
`<route>`» · «канонизируй страницу `<route>`».

**Задача.** Перевести **одну** существующую страницу на публичный барель
`@/components/ds`, **сохранив контент и порядок секций**. Это **верный рефактор, а
не редизайн**: версии / поверхности / цвета / переходы и копирайт НЕ меняются —
меняется только **происхождение блока**: локальный/инлайн/`Lab*` → барель.

**Главный инвариант — верность.** Контент (копирайт, слоты, данные), порядок и
смысл секций сохраняются 1:1. Скилл НЕ красит, НЕ выбирает `version`, НЕ ставит
переходы — это шаг 0 **перед** арт-дирекшном.

**Маппинг секции** — одно из трёх:
- **already-ds** — уже из бареля → не трогаем (только проверяем импорт-путь).
- **swap** — локальный/инлайн/`Lab*` с барель-эквивалентом → подмена + маппинг
  «контент → пропсы» 1:1.
- **gap** — архетип без барель-эквивалента (`component-api-gap`/`Lab*`) → честно
  пометить и маршрутизировать: (a) извлечь сейчас через `redraw-block` (режим B),
  (b) оставить `Lab*` с явным `// TODO ds-gap`, (c) ближайший `ready` DS — только
  с согласия. **Сырой `Lab*` без TODO молча не остаётся.**

**Гейт.** Таблица маппинга «секция → DS-компонент» + решения по гэпам **до**
перезаписи.

**Выход.** Страница, где каждый блок = DS-компонент (или помеченный гэп), готовая
к ручному арт-дирекшну (версии + переходы) и к `page-skin` / `page-motion`.

**НЕ делает:** не batch (это `build-pages`), не раскрашивает, не куёт секции с
нуля. Одна страница за вызов. Маршрутизирует гэпы в `redraw-block` / `component-forge`.

## 3. `retype-section` — свап типа секции с переносом контента

**Триггер:** `/retype-section` · «поменяй тип секции `<N>` на `<route>`» · «секцию
3 из бенто в галерею» · `retype <route>#<секция> → <Type>`.

**Задача.** Сменить **тип** одной секции на другой архетип и **смаппить её
контент** в новый DS-компонент, ничего не теряя и не выдумывая. Направленный свап:
user указывает страницу + секцию + целевой тип.

**Главный инвариант — фиделити контента.** Каждый смысловой слот источника либо
**смаплен** в слот цели, либо помечен **drop** (с согласия user), либо **смёржен**.
Каждый обязательный слот цели либо **заполнен** из источника, либо помечен **needs
value**. Молча терять или сочинять контент нельзя.

**Карта маппинга** строится по семантическим ролям, не по именам вслепую:
- **Шапка**: eyebrow→eyebrow, title→title, sub→sub (обычно 1:1).
- **Коллекция**: элементы источника → элементы цели по под-полям
  (`tile.title→item.title`, `tile.miniTag→item.tag` и т.п.).
- **Особые слоты**: чего нет у цели (Bento `feature`-плитка, медиа-слот) → **drop**
  с пометкой деградации; обязательное у цели, но нет в источнике (Gallery
  `laneLabel`) → **needs value** (предложить дефолт, не подставлять молча).

**Гейт.** Карта маппинга «слот-источник → слот-цель» + флаги (drop / needs-value)
**до** перезаписи.

**Это свап типа, НЕ редизайн.** Целевой компонент берётся как есть (его
дизайн/версии не трогаются), `surface` и позиция секции сохраняются. Цель — только
готовый компонент бареля; если целевой архетип — гэп (`Lab*`), он сначала
извлекается через `redraw-block` (режим B).

**НЕ делает:** не редизайнит (это `redraw-block`), не конвертит всю страницу (это
`ds-ify-page`). Одна секция за вызов.

## 4. Как трио складывается с остальными

Естественный порядок работы над страницей:

```
1. ds-ify-page      — выпрямить страницу до канона @/components/ds (рефактор, без смысловых правок)
   (гэпы → redraw-block / component-forge)
2. retype-section   — при необходимости сменить тип отдельной секции (свап + перенос контента)
3. zone-background  — объединить группы идущих подряд секций под общий фон (тональные зоны)
4. ручной арт-дирекшн / page-skin → page-motion  — версии блоков, цвет, переходы между зонами
```

- **`ds-ify-page`** — шаг 0: канонизирует происхождение блоков. Делать **до**
  любой раскраски/переходов.
- **`retype-section`** — когда тип секции неправильный по смыслу (не «как
  нарисовано», а «не тот архетип»).
- **`zone-background`** — когда несколько секций должны читаться под **одним**
  фоном (тональная зона). Применяется уже на канонизированной странице.
- **`redraw-block`** — общий «руки» для всех трёх: извлекает гэп-архетип в чистый
  DS-компонент или редизайнит один блок до продуктового качества.

Большие оркестраторы (слой A) — когда страница собирается/перебирается целиком:
- **`page-skeleton` → `page-skin` → `page-motion`** — расслоённая сборка: структура
  в нейтрале → раскраска (surface + бренд-цвет) → переходы и motion.
- **`page-composer`** — глубокая реконструкция одной страницы из `ready`-библиотеки
  с одним Design Review Gate.
- **`build-pages`** — batch нескольких страниц из продуктовых брифов (единственный
  скилл, который коммитит и пушит).
- **`page-art-director`** — конкурентный арт-дирекшн всей страницы (два движка →
  судья), слой НАД page-skin/page-motion.

## 5. Рейлы (общие для трио)

- **Один user-гейт** до перезаписи; **одна** единица за вызов (зона / страница /
  секция).
- **Инвариант контента**: `ds-ify-page` и `zone-background` не меняют контент/
  порядок вовсе; `retype-section` переносит контент 1:1 (спорное — на гейт, ничего
  молча не теряется и не выдумывается).
- **Публичный API — только `@/components/ds`**; гэпы — через `redraw-block` или
  явный `// TODO ds-gap`, никогда молча `Lab*`.
- **Контейнер несёт `section-lab ds`; `<ScrollFX/>` смонтирован** — обязательное
  условие (иначе стили схлопываются, флип/reveal не оживают).
- **Фон ≠ проп секции**; `surface` = только тон. Готовые фоновые пресеты — из
  `ds.css`; нужного нет → `visual-layer-forge`.
- Не коммитить/push/PR/`.env`; dev-сервер не поднимать; `pnpm build` — с разрешения.
