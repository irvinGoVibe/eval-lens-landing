# Page Design Patterns — Apple-grade

← [[index|Wiki]] · [[design-system|Design System]]

Как собирать **внутренние страницы**, чтобы они читались как Apple product pages:
ритмичные, разнокалиберные, со scroll-driven появлением, а не плоская сетка
карточек. Это рабочий справочник для skill `build-pages` и `developer-agent`.
Дополняет [[design-system|Design System]] (токены, классы) — здесь про
**композицию, ритм и движение**.

> Правило №1: движение — только через единый rAF-цикл. Главная использует
> [ScrollOrchestrator.tsx](web/src/components/ScrollOrchestrator.tsx);
> **внутренние страницы монтируют [ScrollFX.tsx](web/src/components/ScrollFX.tsx)**
> (один раз, после `<Footer/>`) и подключают движение через data-атрибуты.
> Никаких per-section `useEffect`, никаких сторонних animation-библиотек.

## Антипаттерн (чего избегать)

То, что вышло плоско на первой версии methodology:

- вся страница — одинаковые `.tcard` в `.trust-grid`, секция за секцией;
- каждый блок одной высоты, один фон, один ритм;
- нет полноэкранных моментов, нет движения, нет «дыхания» между блоками;
- текст появляется весь сразу, статикой.

Apple так не делает. Их страницы = **чередование масштабов и режимов**: огромный
тихий hero → полноэкранный statement → плотный блок фактов → галерея → pinned
демонстрация на несколько экранов → спокойный финал.

## Движок (data-атрибуты)

`<ScrollFX/>` (смонтированный на внутренней странице) поднимает их в одном
rAF-цикле с cleanup (безопасно при client-side навигации):

| Атрибут | Что делает | CSS-выход |
|---|---|---|
| `data-reveal` | появление при входе во вьюпорт | класс `.is-in`; варианты `="fade\|up\|left\|right\|scale"`; задержка `style="--reveal-delay:120ms"` (стаггер) |
| `data-scrub` | прогресс прохода элемента через вьюпорт | пишет `--scrub` 0→1 (parallax, заполнение ring, скраб медиа) |
| `data-pin` + `data-pin-steps="N"` | высокий трек + липкая сцена внутри (`data-pin-stage`); шаги загораются по очереди | пишет `--pin` 0→1; на `data-pin-step` — `.is-active` (накопительно), `.is-current` (текущий), `--pin-step` (индекс) |

Базовые стили примитивов — в `globals.css` (секция «Generic scroll-FX
primitives»), reduced-motion обработан там же и в движке. **Не забудь смонтировать
`<ScrollFX/>` на странице** (после `<Footer/>`) — иначе `data-reveal`-элементы
останутся `opacity:0` (пустые секции). Конкретную раскладку
секции (grid, sticky-высоту трека, медиа) страница задаёт своими классами под
обёрткой страницы (напр. `.methodology …`) на **канонических токенах**.

## Плейсхолдеры (видимые)

Везде, где по смыслу нужна **картинка, видео или визуализация смысла**, но
реального ассета нет — ставь **видимый размеченный плейсхолдер** `.media-ph`
(глобальный примитив в `globals.css`), а не пустой/серый div. Цель: с одного
взгляда видно, *куда* и *что* вставлять.

```jsx
<figure className="media-ph" style={{ ["--ratio"]: "16/9" }}
        role="img" aria-label="Hero lens illustration">
  <span className="media-ph__label">Image · hero lens · 16:9</span>
  <span className="media-ph__hint">Lens-gradient violet→cyan→aqua over
    Apple-neutral, soft depth, calm — see prompt in file header</span>
</figure>
```

Правила плейсхолдера:
- **Лейбл** — что это (`Image` / `Video` / `Diagram`), о чём, и **соотношение
  сторон** (`16/9`, `4/3`, `1/1`); размер задаётся `--ratio`, чтобы реальный
  ассет лёг без сдвига лейаута.
- **Hint** — короткий промпт/описание, чтобы было ясно, что генерить.
- Полный промпт — в комментарии-шапке `page.tsx` (§Изображения брифа).
- На тёмных полосах (`.ink`) плейсхолдер автоматически перекрашивается.
- Плейсхолдер — это **не замена** контента секции: текст/смысл рядом должен быть
  настоящим; плейсхолдер только под визуал.

## Каталог блок-архетипов

Собирай страницу из РАЗНЫХ архетипов, не повторяя один. Минимум 5–6 разных на
страницу из 8 секций.

1. **Statement hero** — крупный тихий заголовок (`clamp(44px,7vw,96px)`),
   lens-акцент на ключевой фразе (`.grad-word`), короткий подзаголовок, один CTA.
   Много воздуха. Часто — тёмный (`.band.ink`) или светлый full-bleed.
2. **Full-bleed statement** — одна мысль на весь экран (`min-height:100svh`),
   большой текст по центру, `data-reveal`. Разделяет плотные блоки, даёт «вдох».
3. **Pinned multi-screen** — одна секция на несколько экранов: `data-pin` +
   липкая сцена, по мере скролла построчно/поэтапно появляется текст или
   загораются узлы (`data-pin-step`). Это «как Apple рассказывает про продукт».
   Идеально для процесса/конвейера, «как это работает», timeline.
4. **Editorial split** — текст слева, визуал/медиа справа (или наоборот),
   чередовать сторону между секциями. `data-reveal="left"` / `="right"`.
5. **Horizontal gallery** — горизонтальная лента карточек со scroll-snap
   (`overflow-x:auto; scroll-snap-type:x mandatory`) для наборов (роли судей,
   use cases). Не вертикальная сетка.
6. **Stat band / counters** — крупные числа (mono, tabular, lens-градиент),
   появляются стаггером; каждое число — с источником-подписью.
7. **Scrubbed visual** — элемент с `data-scrub`: parallax-сдвиг, заполнение
   confidence-ring по `--scrub`, прокрутка медиа.
8. **Bento overview** — разнокалиберные тайлы (как на главной), один hero-тайл +
   несколько равных, для обзора возможностей.
9. **Quiet CTA band** — финальный спокойный призыв на контрастной полосе
   (`.band.ink` / `.band.soft`) + pill-кнопка.

## Композиция и ритм

- **Чередуй режимы поверхности:** светлая → тёмная (`.band.ink`) → светлая.
  Минимум один тёмный statement на страницу — даёт «кино».
- **Чередуй масштаб:** full-bleed момент → плотный блок → снова воздух. Не три
  плотных блока подряд.
- **Чередуй раскладку:** editorial-split меняет сторону; gallery горизонтальна
  среди вертикальных секций.
- **Один акцент на экран** (правило design-system): lens-градиент на числах/
  ключевой фразе, не везде.
- **Max-ширина** контента 1180–1200px (`.wrap`/`--maxw`); full-bleed-секции
  тянутся на 100vw, но текст внутри держит max-ширину.
- **Spacing крупный:** секции дышат (`padding: clamp(64px,10vw,160px) 0`),
  Apple-страницы щедры на воздух.
- **Визуальная плотность (Apple-style):** не «текст-стена». На страницу —
  **≥1 горизонтальная галерея** и **несколько визуальных слотов** (картинка /
  диаграмма / визуализация смысла рядом с ключевыми блоками). Если реального
  ассета нет — видимый `.media-ph` (см. «Плейсхолдеры»). Числа/процессы/сравнения
  показывай **визуально** (ring, бары, pipeline, bento), а не только словами.
  Каждая «смысловая» секция заслуживает либо визуализации, либо галереи.

## Типографика в деле

- Hero/statement — `clamp(44px,7vw,96px)`, `line-height:1.02`,
  `letter-spacing:-.03em` (как `.hero .title` / epic-заголовки главной).
- Секционный заголовок — `clamp(33px,5vw,56px)`; на тёмном крупнее.
- Lens-акцент фразы — `.grad-word` (на внутренней странице нужен свой
  `<wrapper> .grad-word` правило с `--lens`, т.к. глобальный `.grad-word` scoped
  под `.hero`).
- Mono-лейблы (`.eyebrow`) — uppercase, трекинг `.16em`, ведущая `.dot`.

## Движение (motion)

- Easing — только `--ease` (замедление, без отскока).
- Reveal — `data-reveal` со стаггером внутри группы (`--reveal-delay` по индексу).
- Pinned-секции — `data-pin`; текст появляется по `.is-active`/`.is-current`.
- Parallax/scrub — `data-scrub` → `--scrub` в `transform`/`background`.
- **reduced-motion** — обязан быть тих: примитивы уже это уважают; кастомные
  эффекты оборачивай в `@media (prefers-reduced-motion:reduce)`.
- **Контент в pin-секциях гейти на `.is-active`, не на `.is-current`.** В
  reduced-motion движок выставляет только `.is-active` (всё показано) и НЕ
  `.is-current` (нет «активного» шага без скролла). Если текст раскрывается
  только по `.is-current` — добавь reduced-motion-оверрайд, иначе контент
  пропадёт у этих пользователей.

## Front-end QA чек-лист (рендеринг)

Помимо «не плоско» — страница обязана **корректно рендериться**. Проверять и
статически (чтение CSS/разметки), и **вживую** (preview): часть дефектов видна
только в браузере. Live-проверка — обязательная часть QA для страниц этого скилла
(подними превью `web`, гоняй `preview_eval`/`preview_resize`/`preview_screenshot`).

### Заголовки и текст
- [ ] Заголовки **не расползаются**: размер через `clamp()`, не фикс-`px`; длинная
  строка не вылезает за контейнер (`overflow-wrap:anywhere`/`hyphens` при нужде).
- [ ] Заголовки/строки **не складываются неожиданно в столбик** по буквам/словам:
  у flex/grid-детей задан `min-width:0` (иначе текст ломает раскладку); нет
  слишком узкой колонки под заголовок.
- [ ] Нет «вдовствующих» одиночных символов/слов из-за жёстких `width`.

### Горизонтальный скролл (строго запрещён)
- [ ] **Нет горизонтального скролла** ни на одной ширине. Частые причины:
  `100vw` при наличии вертикального скроллбара, фикс-ширины, негативные
  margin'ы, медиа без `max-width:100%`, грид/флекс-дети без `min-width:0`,
  абсолют за пределами `overflow`.
- [ ] Проверка вживую: `document.documentElement.scrollWidth <=
  document.documentElement.clientWidth` (через `preview_eval`) на mobile/tablet/
  desktop. Любое превышение — баг.
- [ ] Горизонтальные галереи скроллятся **внутри своего контейнера**
  (`overflow-x:auto`), а не двигают страницу.

### Адаптив
- [ ] Проверить **mobile (375)**, **tablet (768)**, **desktop (1280)**
  (`preview_resize`): секции переформатируются, ничего не налезает/не
  обрезается, tap-таргеты ≥ ~40px.
- [ ] Split-секции на узком — в один столбец без поломки; pinned-секция на mobile
  не «залипает» некорректно (проверить высоту трека/`100svh`).
- [ ] `svh`/`dvh` вместо `vh` для полноэкранных, чтобы мобильные адресные бары не
  ломали высоту.

### Анимация / производительность
- [ ] **Анимации не отстают от экрана**: reveal/scrub/pin двигаются вместе со
  скроллом, без рывков и «догоняния». Анимируем только `transform`/`opacity`
  (compositor), **не** `width/height/top/left/margin` (layout-thrash).
- [ ] Скролл-обработчики — `{passive:true}`, работа батчится в один rAF (движок
  `ScrollFX` это уже делает; кастом не должен делать тяжёлую работу на каждый
  `scroll`).
- [ ] `will-change` — только на реально анимируемых узлах, не на всём подряд.
- [ ] Нет layout-shift при появлении медиа: у слотов задан `aspect-ratio`
  (плейсхолдер `.media-ph` это держит).
- [ ] Live: `preview_console_logs` чист (нет ошибок/варнингов гидрации).

### Корректность DOM/движка
- [ ] `<ScrollFX/>` смонтирован (иначе `data-reveal` → пустые секции — типовой баг).
- [ ] `data-pin` имеет внутри `data-pin-stage` и детей `data-pin-step`; `data-scrub`
  на нужном элементе; варианты `data-reveal` валидны.
- [ ] reduced-motion: контент **виден** (ничего не спрятано за `.is-current`).

### Как чинит fix-loop
Найденные дефекты — это **проваленные пункты** для fix-loop: developer правит CSS
(`min-width:0`, `clamp()`, `max-width:100%`, убрать `100vw`/фикс-ширины, перенести
анимацию на `transform/opacity`, задать `aspect-ratio`), пере-QA вживую до чистого
результата. Не «замазывать» `overflow-x:hidden` на body — искать причину.

## Чек-лист «не плоско ли»

- [ ] ≥5 разных архетипов на 8 секций (не одна сетка карточек)
- [ ] ≥1 pinned multi-screen секция со scroll-driven появлением
- [ ] ≥1 тёмный full-bleed statement (смена режима)
- [ ] чередование светлых/тёмных полос и масштаба
- [ ] **≥1 горизонтальная галерея** + несколько визуальных слотов
  (картинка/диаграмма/визуализация смысла), а не текст-стена
- [ ] каждый нужный визуал — **видимый `.media-ph`** с лейблом и `--ratio`
  (если реального ассета нет), чтобы было ясно, куда вставлять
- [ ] движение через data-атрибуты + смонтированный `<ScrollFX/>`, не useEffect
- [ ] reduced-motion тих; только канонические токены/классы
