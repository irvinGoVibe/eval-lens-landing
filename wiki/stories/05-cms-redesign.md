# Story 05 - CMS Redesign (дизайн-система, английский интерфейс, drag-and-drop загрузка)

Status: Draft
Owner:
Team: —
Priority:
Linear Mapping: Project by default

> **Редизайн CMS поверх реализованной Story 02.** Story 02 подняла рабочую админку (вход под паролем, CRUD статей и репостов, выбор статей для блока на главной, Markdown-редактор, загрузка медиа). Story 05 не меняет данные и функциональность — она переводит визуальный слой админки на дизайн-систему продукта (светлая тема, фиолетовый акцент, SF Pro, hairline-границы), локализует весь интерфейс CMS на английский и улучшает UX загрузки изображений (drag-and-drop + превью в формах статей, file-upload в формах репостов). **Story 05 не реализуется без Story 02.**

## Summary

- **Что меняем.** (1) Полностью переписываем `web/src/app/admin/admin.css` с захардкоженной тёмной палитры (#0f1115/#16191f, синий primary #3b6df0, системный шрифт) на токены дизайн-системы (`--bg #ffffff`, `--violet #6c4cf1`, SF Pro, hairline-границы, мягкие фиолетовые тени), чтобы CMS читалась как продуктовая апка. (2) Переводим весь интерфейс CMS на английский (лейблы, навигация, кнопки, сообщения; сейчас смешано RU/EN). (3) Улучшаем UX загрузки: в формах статей — зона drag-and-drop с превью, индикатором прогресса и понятными ошибками; в формах репостов — добавляем file-upload для cover/video/photos (сейчас только ручной ввод URL).
- **Зачем.** Текущий дизайн CMS «вышел плохо», стоит вне дизайн-системы и противоречит approved-политике («внутренние страницы строятся на тех же токенах»). Интерфейс смешан по языку. Загрузка изображений неудобна (вставка `![](url)` в конец тела без превью; в репостах вообще ручной URL).
- **Эффект.** Редактор работает в единообразном продуктовом UI на английском; изображения грузятся перетаскиванием с превью и прогрессом; репосты больше не требуют ручного копирования URL. Функциональность, данные и схема не меняются; публичная часть блога без регрессий; секреты остаются server-side.

## Problem

- `admin.css` помечен в коде как «not part of the public visual system» — намеренно отдельная тёмная палитра, синий primary, системный шрифт. Это прямой конфликт с approved `design-system.md`: «Внутренние страницы сайта строятся на этих же токенах, чтобы лендинг и продукт читались как одно целое». Расхождение осознанное, но больше не оправдано — пользователь хочет, чтобы CMS была похожа на продуктовую апку.
- Интерфейс CMS смешан по языку: бренд и категории — EN, лейблы и навигация — RU («Заголовок», «Дек», «Рубрика», «Обложка», «Сохранить», «Загрузить медиа», «Блог»/«Главная»). i18n-инфраструктуры в проекте нет.
- UX загрузки изображений слабый: единственный image-upload в `MarkdownEditor` («Загрузить медиа» → вставка `![](url)`/`::video{src=}` в конец тела, без превью/структуры/прогресса); обложка статьи — через URL-поле или голый `<input type=file>`; репосты (`LoopForm`) — только ручной ввод URL для cover/video/galleries. `cms.md` подразумевает загрузку медиа репостов в Storage, но `LoopForm` этого не умеет — разрыв doc vs code.

## Goal

- Админка визуально следует дизайн-системе продукта: светлая база `--bg #ffffff`, фиолетовый primary `--violet #6c4cf1` (вместо синего #3b6df0), SF Pro/Menlo, hairline-границы, мягкие фиолетовые тени; компоненты (кнопки primary/ghost, inputs inset, chips/tags, status-chips) и раскладки соответствуют `design-system.md`.
- Весь интерфейс CMS — на английском: навигация, лейблы форм, кнопки, состояния, сообщения об ошибках/успехе.
- В формах статей загрузка изображений — drag-and-drop с превью загруженного, индикатором прогресса и понятными сообщениями об ошибках.
- В формах репостов появляется file-upload для cover/video/photos (помимо/вместо ручного ввода URL).
- Весь медиа- и admin-функционал остаётся строго server-side (service_role, server-only); drag-and-drop UI — клиентский, но сам upload идёт через server action.
- Публичная часть блога (рендер статей, `.article-*` превью в редакторе) не регрессирует.
- Смена политики «CMS теперь следует дизайн-системе» зафиксирована в `cms.md`/`design-system.md`.

## Scope

### In scope

- **Визуальный переписк `web/src/app/admin/admin.css`** — замена захардкоженной тёмной палитры (#0f1115/#16191f, синий #3b6df0, системный шрифт) на токены дизайн-системы (light-тема: `--bg #ffffff`, `--bg-soft #f5f5f7`, `--violet #6c4cf1`, `--fg #1d1d1f`, `--muted`, `--border`/`--border-2`/`--hairline`, SF Pro/Menlo, тени `--sh-1/2/3`, радиусы `--r-xs/sm/md/lg/pill`, `--ease`). Все `.admin-*` классы приводятся к компонентам дизайн-системы: кнопки primary violet / ghost, inputs inset, chips/tags, status-chips; раскладка по паттернам rail+stage / split / bento где уместно.
- **Локализация интерфейса CMS на английский** — все строки в admin-зоне: навигация (`AdminNav` — Blog/Home и вложенные Articles/Reposts, Blog block), лейблы форм (`ArticleForm`: Title/Deck/Category/Cover; `LoopForm`: cover/video/photos/поля), кнопки (Save, Upload media и т.п.), сообщения об ошибках/успехе, тексты `LoginForm`/`LogoutButton`/`FeaturedPicker`/`DeleteButton`, заголовок shell. Hardcode EN-строк в admin-зоне (i18n-инфраструктуры нет — допустимо).
- **Drag-and-drop + превью в формах статей** — клиентская зона перетаскивания с: превью загруженного изображения, индикатором прогресса загрузки, понятными сообщениями об ошибках (тип/размер/сбой). Применяется к обложке статьи и к вставке изображений тела в `MarkdownEditor` (замена нынешней кнопки «Загрузить медиа» с вставкой `![](url)` в конец). Upload выполняется через существующие server actions (`uploadArticleMediaAction`, `uploadMedia`), не напрямую из клиента.
- **File-upload для репостов** — в `LoopForm` добавить загрузку файлов для cover, video и galleries (через `uploadMedia(file, 'bento'|'photos'|'video')`), сохранив возможность ручного ввода URL как фолбэк. Закрывает разрыв «cms.md упоминает медиа репостов в Storage, а LoopForm только URL».
- **Server-side инвариант** — весь upload остаётся через server actions под `service_role` (`web/src/lib/cms/storage.ts`, `admin-client.ts` — server-only); DnD-компоненты — клиентские, но секреты в клиентский бандл не попадают.
- **Консистентность превью Markdown** — превью тела в `MarkdownEditor` (через `MarkdownBody` на `.article-*`) остаётся визуально консистентным после редизайна admin-обёртки; публичные `.article-*` классы не ломаются.
- **Патч документации** — отметить в `cms.md`/`design-system.md` смену политики: CMS теперь следует дизайн-системе (светлая тема), интерфейс на английском, file-upload медиа репостов реализован.

### Out of scope

- **Медиа-библиотека / браузер бакета** — листинг всего бакета `media`, переиспользование существующих объектов, удаление объектов Storage из UI. Story добавляет только upload-в-форме (DnD + превью), не файловый менеджер.
- **Новые поля данных, изменения схемы БД, миграции** — данные и схема Story 01/02 не трогаются.
- **Изменение функциональности CMS** — CRUD-логика, auth, переключатель `status`, выбор/порядок статей для блока на главной, Markdown-рендер и директивы остаются как в Story 02; меняется только визуал, язык интерфейса и upload-UX.
- **Многопользовательская модель / роли / личные кабинеты** — остаётся один общий пароль.
- **Редактирование статичных секций главной** (Hero/Problem/OrangeGlow/Workflow/Decisions/Bento) — вне CMS, как и было.
- **i18n-инфраструктура** (переключатель языков, словари, локали) — не заводим; строки CMS hardcode EN.
- **Публичная часть блога** — `app/blog/*`, `components/blog/*`, публичные классы `globals.css` не редизайнятся (только не должны регрессировать).
- **Изменение порта dev (3005), правила ScrollOrchestrator** — без изменений.

## Users / Actors

- primary user: редактор контента — входит в CMS, ведёт статьи и репосты, грузит изображения перетаскиванием; теперь видит единый продуктовый светлый UI на английском.

## Main Flow

1. Подтверждается смена политики (CMS следует дизайн-системе) и подход к стилям (миграция `admin.css` на токены/утилиты vs scoped CSS на переменных продукта).
2. Переписывается `admin.css`: тёмная палитра → токены дизайн-системы (светлая тема, violet primary, SF Pro, hairline, мягкие фиолетовые тени, канонические радиусы/easing); все `.admin-*` классы приводятся к компонентам дизайн-системы.
3. Все строки интерфейса CMS переводятся на английский (навигация, лейблы, кнопки, сообщения) — hardcode в admin-компонентах.
4. В формах статей реализуется drag-and-drop зона с превью, прогрессом и обработкой ошибок; upload идёт через существующие server actions (`uploadArticleMediaAction`/`uploadMedia`).
5. В `LoopForm` добавляется file-upload для cover/video/photos через `uploadMedia(file, folder)`; ручной ввод URL сохраняется как фолбэк.
6. Проверяется server-side инвариант: секреты и admin-клиент не в клиентском бандле; DnD-компоненты клиентские, upload — server action.
7. Проверяется консистентность превью Markdown и отсутствие регрессий публичного блога.
8. Патчатся `cms.md`/`design-system.md` (смена политики, EN-интерфейс, file-upload репостов).
9. Проверка статикой/диффом; `cd web && pnpm build` — **только по явной просьбе user**.

## Business Rules

- Визуальный слой CMS строится только на токенах дизайн-системы (`design-system.md`); новые цвета/тени/радиусы/шрифты не выдумываются. Тени — мягкие фиолетовые (`--sh-1/2/3`), не серый drop-shadow. Primary-действие — violet, не синий.
- Не плодить новый локальный «диалект» стилей: либо миграция `admin.css` на канонические токены/Tailwind-утилиты, либо scoped CSS на CSS-переменных продукта — единый подход, выбранный в Open Questions, без смешения.
- Все строки интерфейса CMS — на английском. i18n-инфраструктуры нет; строки hardcode в admin-зоне (это приемлемо для admin, не публичная часть).
- Весь upload медиа выполняется через server actions под `service_role`; `admin-client.ts`/`storage.ts` остаются server-only. DnD-UI клиентский, но секреты в клиентский бандл не попадают (инвариант CLAUDE.md/entrypoint).
- Загрузка в форме статьи — drag-and-drop с превью, прогрессом и понятными ошибками. Загрузка в репостах — file-upload в правильные подпапки (`bento` для cover/галереи, `photos`, `video`), с сохранением ручного URL как фолбэка.
- Это редизайн/UX, не функциональность: CRUD, auth, status-переключатель, выбор статей для главной, Markdown-рендер и директивы не меняются по поведению; схема БД и поля данных не трогаются.
- Превью Markdown (`.article-*`) остаётся консистентным; публичные классы блога не ломаются.
- Markdown-редактор и DnD-компоненты держатся в admin-зоне и не утекают в публичный бандл.
- Новые зависимости (если нужен dropzone-пакет) — только по согласованию; предпочтение — нативный HTML5 drag-and-drop без новых пакетов. Только pnpm; `package-lock.json` в `web/` не создавать.
- Порт dev 3005 не менять; анимации — через `ScrollOrchestrator` (для admin-зоны не дробить на per-section useEffect, если вообще понадобятся).

## Edge Cases

- Перетащен файл неподдерживаемого типа (не изображение/видео) — понятная ошибка, загрузка не стартует.
- Файл превышает допустимый размер / сбой загрузки — понятная ошибка, без частичной записи; форма не теряет уже введённые данные.
- Перетаскивание нескольких файлов в зону — поведение зафиксировать (один файл / последовательность); для галереи репоста — допустимо несколько.
- Повторная загрузка cover/media поверх существующего (`uploadMedia` `upsert:false`, имя = slug + Date.now()) — создаётся новый объект, старый осиротевает в бакете (чистка вне scope; зафиксировать поведение).
- Прерванная/медленная загрузка — индикатор прогресса отражает состояние; отмена/повтор — поведение зафиксировать при реализации.
- Markdown-превью: после смены темы admin-обёртки `.article-*` превью не должно унаследовать admin-цвета и должно остаться консистентным публичному рендеру.
- Светлая тема и читаемость: status-chips/семантика (green/amber/violet/muted) должны остаться различимы на светлом фоне (контраст по дизайн-системе).
- EN-локализация не должна сломать существующие категории (Press Release/Product/Research уже EN) и значения данных — переводятся только UI-строки, не контент/значения в БД.
- DnD не поддерживается / отключён JS — должен оставаться фолбэк (обычный file-input и/или ручной URL), чтобы загрузка не блокировалась.

## Open Questions

- **Подход к стилям admin** — миграция `admin.css` на канонические токены/Tailwind-утилиты дизайн-системы vs переписать `admin.css` на CSS-переменные продукта (scoped под `.admin-root`). Выбрать единый подход при реализации, не плодить новый «диалект».
- **Dropzone-зависимость vs нативный HTML5 DnD** — нужен ли пакет (react-dropzone) или хватит нативного drag-and-drop без новых зависимостей. Предпочтение — нативный (CLAUDE.md: новые пакеты только по согласованию).
- **Осиротевшие объекты Storage** — при повторной загрузке cover/media (`upsert:false`, имя = slug+Date.now()) старый объект остаётся в бакете. Чистка/удаление объектов — вне scope; зафиксировать поведение, не решать здесь.
- **Достаточность референса** — пиксельного маппинга из внешнего стайл-гайда/репо продукта нет; строим только по `design-system.md` (+ `page-design-patterns.md`). Подтвердить, что этого достаточно для приёмки «похоже на продуктовую апку».
