# Process

← [[index|Wiki]]

Один документ про то, как устроена работа в проекте: слои, документы, статусы, рабочий ритм и правила AI-ассистента.

## 1. Слои проекта

Поток знаний идёт так:

`notes/ -> wiki/ -> Linear -> src/`

- **`notes/`** — входящий буфер: сырые мысли, баги, идеи, дневные заметки (`daily/`), дневные отчёты (`done/`). Append-only: прошлые дни не переписываются. Типы сигналов — `notes/README.md`.
- **`wiki/`** — источник правды для людей и AI:
  - `product/` — что строим, для кого, какая ценность
  - `architecture/` — как система устроена, данные, доменные правила
  - `stories/` — story-спеки: раскладка фич на задачи с acceptance criteria (шаблон — `_template.md`)
  - `decisions/` — ADR: почему выбран подход (шаблон — `_template.md`)
  - `processes/` — этот документ, [[agent-workflow|Agent Workflow]], [[linear-workflow|Linear Workflow]]
- **`Linear`** — execution tracking: projects, issues, статусы. Опционален; если не подключён, трекингом служат stories. Linear не хранит смысл продукта — содержание задач опирается на `wiki/`.
- **`src/`** — только реализация. Продуктовая правда сначала ищется в `wiki/`, а не угадывается по коду.

### Как искать источник

- вопрос про смысл → `wiki/product/`, `wiki/architecture/`
- точный спек задачи → `wiki/stories/`
- почему так решили → `wiki/decisions/`
- сырые идеи, история → `notes/`
- статус исполнения → `Linear` (или story-файл)
- фактическая реализация → `src/`

### Как вносить изменения

- новая мысль → `notes/daily/` (`/note`)
- сформировалось понимание → `wiki/product/` или `wiki/architecture/`
- готово к исполнению → story в `wiki/stories/` (`/story-draft`), опционально публикация в Linear (`/linear-publish`)
- взято в разработку → `src/` (`/implement-story`)

Базовый путь не жёсткий: задача может идти `notes -> Linear` напрямую, если контекст ещё не дозрел до документации.

## 2. Формат документа

Каждый документ `wiki/` начинается с frontmatter:

```yaml
---
title: <Читаемое название без двоеточий (или в кавычках)>
status: draft
version: 1.0
updated: YYYY-MM-DD
---
```

Правила:

- **Имена файлов** — lowercase, kebab-case, без пробелов и подчёркиваний, без дублирования имени папки.
- **Ссылки** — короткая wiki-форма `[[name|Display]]` (Obsidian-совместимо).
- **Back-link** `← [[index|Wiki]]` в начале документа; для корневых допустим `← [[Home]]`.
- **Один документ — один вопрос.** Смысл не дублируется между документами; ставится ссылка.
- Новый документ создаётся копией `wiki/_template.md`.

## 3. Статусы документов

`draft → generated → review → approved → locked` (+ пассивные `deprecated`, `archive`)

- **Человек** создаёт документ со `status: draft`.
- **AI** при создании или содержательной правке ставит `status: generated` («AI-output, требует ревью») и атомарно, в одном редактировании: повышает `version`, обновляет `updated` на сегодня.
- **Версия — только `x.y`**: minor для правок внутри структуры, major для переписывания или смены смысла. Формат `x.y.z` запрещён.
- `review`, `approved`, `locked` ставит **только человек** после проверки. AI никогда не повышает статус сам.
- **`locked`** AI не правит без явного подтверждения: остановиться, показать путь/статус/что изменится, задать явный вопрос, дождаться явного «да».
- **`deprecated` / `archive`** — не правятся и не используются как источник истины.
- Косметика (опечатки, форматирование) — исключение: статус и версия не меняются.

## 4. Рабочий ритм

| Когда | Что | Команда |
|---|---|---|
| В течение дня | Фиксировать сигналы: баги, идеи, наблюдения | `/note [prefix:] <text>` |
| Конец дня | Структурированный отчёт: сделано / в процессе / блокеры | `/done` |
| Раз в несколько дней | Перенести дозревшие темы из notes в wiki | `/curate` |
| Фича готова к спеку | Черновик story | `/story-draft <task>` |
| Story одобрена | Публикация в Linear (если подключён) | `/linear-publish` |
| Story в работу | Оркестрация реализации через агентов | `/implement-story` |
| Реализация принята | Закрытие: отчёт, merge, очистка | `/finish-story` |

Промпт каждой команды — `.claude/commands/<name>.md`; AI читает его перед выполнением и следует без отклонений.

## 5. Правила AI-ассистента

Главное правило: **AI не выдумывает недостающую проектную логику.** Если логика не зафиксирована — сначала оформить её в правильном слое, потом использовать.

Порядок чтения контекста: `wiki/` → `Linear` → текущий чат → `notes/` → `src/`.

AI запрещено:

- создавать engineering-задачи без source docs в `wiki/`
- писать новый продуктовый смысл сразу в Linear
- считать код единственным источником истины при наличии документации (расхождение кода и доков отмечается явно, «удобная» версия молча не выбирается)
- создавать vague-задачи без deliverable и acceptance criteria
- плодить дубликаты задач без проверки существующих
- придумывать статусы, поля, workflow и slash-команды, которых нет в проекте

Чеклист перед созданием или правкой чего-либо:

- В каком слое должна жить эта информация?
- Найден ли source of truth?
- Нет ли дубля документа или задачи?
- Можно ли показать, откуда взято решение?
