# Linear Workflow

← [[index|Wiki]]

Фиксирует, как EvalLense Landing работает с задачами в Linear.

Главная цель:
- убрать хаос
- разделить product thinking и engineering execution
- сделать единый путь от документации до реализации
- не допускать задач без нормального source context

---

## 0. Если Linear не подключён

Linear — **опциональный** слой. Если при bootstrap проект не подключил Linear:

- слой execution-трекинга пропускается целиком
- story-файлы в `wiki/stories/` остаются единственным трекингом задач — статус задачи виден по frontmatter story и Change Log
- все Linear-команды (`/linear-publish`, Linear-ветки `/note` и `/task`) спят: они сообщают, что Linear не подключён, и не делают ничего
- агент `linear-drafter` и все auto-transitions из §7b не используются
- весь остальной flow (`/story-draft`, `/implement-story`, `/finish-story`) работает без изменений

Остальные разделы этого документа актуальны только при подключённом Linear.

---

## 1. Роль Linear в системе

Linear в EvalLense Landing нужен не для хранения смысла продукта, а для управления исполнением.

Источник правды по смыслу:
- `notes/` — сырые находки
- `wiki/` — человекочитаемая логика, стандарты и stories

Источник правды по исполнению:
- `Linear`

Значит:
- продукт не живёт в Linear
- архитектура не живёт в Linear
- в Linear живут только execution-объекты

---

## 2. Команды в Linear

В системе две команды:

- `—`
- `—`

### —
Отвечает за:
- discovery
- формализацию идеи
- scope
- постановку задачи
- документацию
- handoff в engineering

### —
Отвечает за:
- техническую декомпозицию
- реализацию
- review
- проверку результата
- обратную связь в документацию при изменении логики

---

## 3. Главный flow

Общий путь задачи:

`notes -> wiki -> — -> — -> src`

Расшифровка:
1. мысль, баг, идея или задача попадает в `notes/`
2. после осмысления оформляется в `wiki/` (при необходимости — как story в `wiki/stories/`)
3. на основе этого создаётся issue в `—`
4. после готовности документации создаётся или передаётся issue в `—`
5. задача реализуется в коде

---

## 4. Какие типы задач бывают

## 4.1 Product issue

Нужна, когда надо:
- исследовать проблему
- определить scope
- описать flow
- зафиксировать решение
- подготовить handoff
- собрать acceptance criteria

Product issue не должна содержать задачу вида:
- просто сделать фичу
- просто поправить backend
- просто сверстать экран

Она должна сначала ответить:
- что делаем
- зачем делаем
- что входит в scope
- что не входит
- как выглядит готовый результат

## 4.2 Engineering issue

Нужна, когда уже понятно:
- что реализуем
- где реализуем
- откуда source truth
- как проверить готовность

Engineering issue должна быть атомарной и исполнимой.

## 4.3 Sync / validation issue

Опциональный тип задач для:
- QA
- проверки acceptance
- валидации, что реализация совпадает со спеком
- синхронизации документации после изменений

---

## 5. Workflow команды —

Рекомендуемые статусы для —:

1. `Inbox`
2. `Discovery`
3. `Scoped`
4. `Documentation Ready`
5. `Ready for Engineering`
6. `Done`

### 5.1 Inbox
Сюда попадает:
- сырая идея
- баг из `/note`
- новое направление
- неразобранный запрос

Критерий выхода:
- стало понятно, стоит ли вообще этим заниматься

### 5.2 Discovery
Здесь происходит:
- анализ проблемы
- сбор контекста
- определение пользователя
- оценка ценности
- первичная формулировка решения

Критерий выхода:
- есть понятная постановка задачи

### 5.3 Scoped
Здесь фиксируем:
- in scope
- out of scope
- dependencies
- риски
- affected areas
- основные документы

Критерий выхода:
- scope больше не плавает

### 5.4 Documentation Ready
Здесь уже должны быть готовы:
- `wiki` источник
- story в `wiki/stories/` при необходимости
- acceptance criteria
- ссылки на связанные материалы
- понятный deliverable

Критерий выхода:
- engineering может брать задачу без угадывания

### 5.5 Ready for Engineering
Это точка handoff.

Смысл:
- product часть завершена
- задача готова к публикации или передаче в engineering
- если нужно, issue уже декомпозирована

### 5.6 Done
Ставим только когда:
- product работа по задаче действительно завершена
- handoff выполнен
- дальше изменений по постановке не требуется

## 6. Workflow команды —

Используем стандартный набор Linear-статусов:

1. `Backlog`
2. `Todo`
3. `In Progress`
4. `In Review`
5. `Done`

Промежуточный «Ready» и отдельный «QA / Validation» не используются — упрощаем pipeline и завязываем переходы на агентский flow (см. §7b).

### 6.1 Backlog
Задача зафиксирована (через `/note`, `/task` или прямо из обсуждения), но source docs и acceptance criteria ещё не оформлены. Story для неё ещё нет.

### 6.2 Todo
Story оформлена и записана в `wiki/stories/`. Acceptance criteria есть, scope зафиксирован, можно брать в работу. Реализация ещё не началась.

### 6.3 In Progress
`developer-agent` активно меняет код по плану `implementation-planner`. Задача выходит из этого статуса либо назад в `Todo` (если orchestrator остановил цикл), либо в `In Review` после QA pass.

### 6.4 In Review
QA pass получен (`qa-acceptance-reviewer` → `ready for manual review`), `release-reporter` вернул финальный отчёт. Ожидает ручной проверки user — UI smoke, code review. AI-агенты сюда задачу двигают, но дальше её **не двигают**.

### 6.5 Done
Ставит **только человек** после ручной проверки. Никакой агент и никакая slash-команда сами в `Done` не переводят.

## 7. Жёсткое правило handoff из — в —

Задачу можно передавать в `—` только если есть всё ниже.

Обязательный минимум:
- есть source doc в `wiki/`
- есть story в `wiki/stories/`, если задача требует технической точности
- зафиксирован scope
- есть deliverable
- есть acceptance criteria
- понятны зависимости
- понятна команда и зона ответственности

Если хотя бы одного пункта нет:
- задача не идёт в engineering
- задача возвращается в `—`

## 7a. Автоматизация: публикация через slash-команды

Linear-объекты в EvalLense Landing заводятся не вручную, а через нейроассистента и Linear MCP. Точки входа:

- `/note [prefix:] <text>` — добавить запись в `notes/daily/<YYYY-MM-DD>.md` и опционально завести Issue. Маршрут зависит от префикса:
  - `bug:` → Issue в **— Backlog**
  - `task:` → Issue в **— Backlog**
  - `idea:` → Issue в **— Inbox**
  - без префикса → только запись в notes, Linear не трогаем
- `/task <text>` — quick capture без записи в notes. Создаёт Issue сразу в **— Backlog** (по умолчанию) или в **— Inbox**, если в тексте явно сказано «idea». Используется, когда задача родилась прямо из обсуждения и в notes писать смысла нет.
- `/story-draft <текст / ссылка>` — оформляет story-черновик. На записи файла команда смотрит в frontmatter story поля `linear_project` и `linear_issues`:
  - если `linear_issues` задан — двигает уже существующий engineering issue из `Backlog` в `Todo`
  - если поле пустое — предлагает создать новый engineering issue сразу в `Todo` с привязкой к story
- `/linear-publish <путь к story>` — публикация story в Linear:
  - крупная story (`Linear Mapping: Project + child issues`) → Linear Project + child Issues. Если в story задан `linear_project` (ID существующего Linear Project — общий контейнер направления), child Issues создаются внутри него, новый Project не заводится.
  - атомарная story (`Issues only`) → один или несколько Issues без отдельного Project.
  - дубли проверяются по `linear_issues` в frontmatter и по title/scope: если уже привязан issue — двигаем его, не дублируем.
- `/implement-story <путь к story>` — оркестратор реализации. Сам автоматически двигает связанный engineering issue по фазам (см. §7b). Никаких дополнительных подтверждений на каждый Linear-переход не запрашивает: подтверждение фазы implementation **= согласие на соответствующий переход**.

Полная таблица slash-команд (включая не связанные с Linear) — в [[process|Process]] §4.

Ручное создание issue через Linear UI допустимо, но не рекомендуется: исчезает трассируемость `notes → wiki → story → Linear`, и ссылки на source-материалы приходится проставлять вручную.

## 7b. Auto-transitions: маппинг агентского flow на Linear

Engineering-issue двигается по pipeline автоматически, на ключевых триггерах команд. Никаких отдельных вопросов «перевести в следующий статус?» не задаём — старт фазы уже подтверждён user.

| Триггер | Кто исполняет | Linear переход | Условие |
|---|---|---|---|
| `/note bug:` / `/note task:` подтверждено | команда `/note` | создание Issue в `—: Backlog` | префикс задан, user сказал «да» |
| `/note idea:` подтверждено | команда `/note` | создание Issue в `—: Inbox` | префикс задан, user сказал «да» |
| `/task <text>` подтверждено | команда `/task` | создание Issue в `—: Backlog` (или `—: Inbox` для idea) | user сказал «да» |
| `/story-draft` записал файл story | команда `/story-draft` | привязанный Issue: `Backlog → Todo` (или создание нового сразу в `Todo`) | story прошла required approvals |
| `/implement-story` Phase 2 (start code) | команда `/implement-story` | привязанный Issue: `Todo → In Progress` | user подтвердил переход в фазу code |
| `/implement-story` Phase 5 (release-reporter после QA pass) | команда `/implement-story` | привязанный Issue: `In Progress → In Review` | `qa-acceptance-reviewer` вернул `ready for manual review` |
| Ручная проверка пройдена | **user, вручную в Linear UI** | `In Review → Done` | агенты этого не делают **никогда** |

Жёсткие инварианты:
- ни один subagent (`developer-agent`, `qa-acceptance-reviewer`, `release-reporter`, `linear-drafter`, `data-security-reviewer`, `domain-logic-reviewer`, `ui-reviewer` и т.д.) не вызывает Linear MCP write напрямую. Subagents возвращают отчёт, переходы делает оркестратор-команда в основном контексте.
- `Done` ставит только user. Если agent или команда видит, что задача «как будто Done» — она оставляет её в `In Review` и пишет это в финальный отчёт.
- если переход не удался (Linear API отказ, статус не существует в команде, issue не найден) — команда пишет ошибку user, остальной flow не блокирует. Story и код продвигаются, Linear синхронизируется потом вручную.
- если в story нет связанного Linear issue (поле `linear_issues` пусто и `/linear-publish` ещё не вызывался) — переходы не делаются, команды просто реализуют свою основную логику. Никаких «теневых» Issues не заводим.

## 8.1 Жёсткое правило — PRODUCT issue без технического жаргона

**Любой issue, публикуемый в team —, формулируется на продуктовом языке. Технические термины запрещены в title и description.** Правило применяется как при ручном создании, так и при автоматической публикации через `/linear-publish`, `/story-draft`, `/note`, любые sub-агенты и slash-команды.

Цель — product-человек должен принять решение, не открывая код, архитектурные решения, миграции, slash-команды и `wiki/architecture/*`.

### Что запрещено в PRODUCT issue

- **Имена таблиц, колонок, индексов, политик доступа к данным**: имена сущностей схемы данных, ограничения уникальности, внешние ключи, каскадные удаления.
- **SQL-фрагменты** и любые фрагменты запросов к данным.
- **Названия API-endpoints, route-paths, файловых путей**: `/api/...`, `src/...`, пути к миграциям. Допустимо обобщение «публичная страница события», «страница входа», «возврат после входа».
- **Английские технические термины**: OAuth, JWT, FK, PK, SSR, middleware, route group, callback, redirect, sanitize, idempotent, race condition. Заменяются на смысл по-человечески («вход через Google», «проверка прав», «приглашение», «возврат на страницу»).
- **ID других Linear issues в основном тексте**. **Исключение**: ссылка на родительский parent issue в одну строку сверху («Связан со story N, родительская задача — <ID>»). Внутри текста — никаких упоминаний ID.
- **Имена wiki-файлов / migration-файлов / story-файлов**. Если нужно сослаться — пишется по-человечески: «правила работы публичной страницы», «продуктовая документация».
- **Code-snippets, конфиг-фрагменты, env-переменные, имена пакетов**.
- **Internal jargon проекта**: внутренние названия модулей, процессов и технических механизмов. Если без понятия не обойтись — объясняется одной фразой в скобках на простом языке, без code-сленга.

### Что обязательно в PRODUCT issue

- **Описание сценария** на языке пользователя: «организатор делает X, участник видит Y».
- **Варианты решения** для каждого нерешённого вопроса — пронумерованы а/б/в, каждый с короткой формулировкой последствия («требует доработки», «текущий рабочий вариант», «блокировка»).
- **Конкретные примеры данных** с фейковыми именами: `имя1@почта.com`, «проект "Осенний фестиваль"», вместо абстрактного `<email>` или выражений из кода.
- **Что и где зафиксировать после решения** — раздел про продуктовую документацию (без указания конкретного пути к .md-файлу).
- **Acceptance criteria** в формате «человек видит X», «текст звучит как Y», «на телефоне читаемо», без отсылок к компонентам, props, политикам данных, миграциям.

### Допустимые исключения

- **Parent issue link** одной строкой в самом верху — единственное место, где можно указать ID. Формулировка: «Связан со story N (родительская задача — <ID>)».
- **Title issue** может иметь префикс по конвенции команды, это не жаргон.
- **Имя продукта/фичи** на естественном языке («публичная страница события», «форма подачи заявки») — это не жаргон, а продуктовое название.

### Кто отвечает

- При ручном создании — автор issue.
- При автоматической публикации (`/linear-publish`, sub-агенты, `/story-draft`, `/note`) — slash-команда обязана применить этот фильтр перед save_issue в — team. Engineering-формулировки из story переводятся в продуктовый язык до публикации, не после.
- При ревью — любой человек, увидевший технический термин в PRODUCT issue, имеет право вернуть на переписку (или переписать сам).

### Engineering и QA issues — правило не действует

В — и QA team технические термины разрешены и ожидаемы — это рабочий язык исполнителя. Правило касается **только** — team.

## 10. Правило атомарности

Одна issue = один проверяемый результат.

Нельзя смешивать в одной engineering issue:
- backend
- frontend
- migration
- analytics
- infra

если это разные deliverables

Нельзя смешивать в одной product issue:
- discovery
- финальный spec
- engineering implementation

если это реально разные фазы

## 11. Story-as-parent-issue (декомпозиция крупной story)

Крупная story (`Linear Mapping: Project + child issues`) публикуется в существующий Linear Project как **один parent issue + N sub-issues**. Цель — не плодить плоский список из 30+ задач в общем контейнере, а держать иерархию по story.

### Структура

- **Parent issue** = «зонтик» story. Один на каждую story.
  - Title: `Story XX — <название story>`.
  - Description: summary + Acceptance Criteria обзорно + ссылка на `wiki/stories/XX-*.md` (permalink репозитория).
  - State после `/linear-publish`: `Todo`. После start implementation parent двигается через `/implement-story` синхронно с sub-issues.
- **Sub-issues** = атомарные implementation / QA / product задачи.
  - Каждый — отдельный deliverable (миграция, backend-модуль, UI, doc-patch, QA-матрица, product acceptance).
  - Создаются с `parentId = <parent id>` и `projectId = <общий Linear Project>`.
  - Team каждого sub-issue выбирается по §2 (— vs —); parent наследует team преобладающей фазы (как правило — — для implementation-stories).

### Done

`Done` ставит human-user отдельно:
- для каждого sub-issue — после его acceptance;
- для parent — после того как все sub-issues в `Done` и story переведена в `approved` / `locked`.

Никакой агент Done не ставит (см. §6.5, §7b).

### Frontmatter story

Помимо `linear_project` и `linear_issues`, story получает:

```yaml
linear_project: <ID общего Linear Project>
linear_parent_issue: <ID parent issue>
linear_issues:
  - <ID sub-issue 1>
  - <ID sub-issue 2>
  - ...
```

Это даёт трассируемость `story → parent → sub-issues` без UI-шаманства.

### Когда parent НЕ нужен

- story атомарна (`Linear Mapping: Issues only`) — публикуется как 1-2 Issue без parent;
- story-черновик (`/story-draft`) до `/linear-publish` — parent не создаётся.

### Пример

- Parent: `Story NN — Публичная страница события` (—, Todo)
  - Sub 1: ENG: миграция — политика публичного чтения опубликованных событий
  - Sub 2: ENG: методы выборки события по публичной ссылке
  - Sub 3: ENG: открытый роут публичной страницы + страница «не найдено»
  - Sub 4: ENG: UI страницы — шапка / даты / статус
  - Sub 5: ENG: режим предпросмотра для организатора

## 12. Что должно быть связано между слоями

Каждая важная задача должна иметь ссылки вверх по цепочке:

- `notes` или исходный capture
- `wiki` источник истины
- related story в `wiki/stories/`
- parent issue или linked issue

Минимум:
- `wiki` ссылка обязательна для product задач
- `wiki` + story обязательны для engineering задач, если задача не тривиальна

---

## 13. Что нужно проверить перед созданием issue

Перед созданием issue нужно проверить:

1. Это точно задача, а не просто мысль?
2. В какой слой её сначала нужно положить: notes, wiki или story?
3. Это задача для — или для —?
4. Есть ли уже похожая issue?
5. Есть ли source docs?
6. Это одна задача или нужна декомпозиция?
7. Можно ли проверить результат?

Если нет ответа хотя бы на половину этих пунктов — issue создавать рано.

---

## 14. Когда задача считается качественно поставленной

Хорошая задача:
- имеет один результат
- опирается на source docs
- не заставляет исполнителя угадывать
- содержит проверяемые acceptance criteria
- имеет понятный scope
- вписана в dependency chain

Плохая задача:
- абстрактная
- без контекста
- без источников
- с несколькими deliverables сразу
- без критерия готовности
